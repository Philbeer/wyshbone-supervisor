# Wyshbone Supervisor Suite

## Overview
Wyshbone Supervisor is a B2B lead generation system designed for automatic prospect identification and scoring. It delivers real-time lead suggestions with contact information via email and an integrated chat. The system aims to enhance sales processes, expand market reach, provide actionable data, and improve workflow efficiency. Future ambitions include integrating AI for plan evaluation, deep research, and intelligent replanning to optimize lead generation and verification.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### UI/UX Decisions
The frontend uses React, TypeScript, Vite, and Wouter. Styling is managed with Tailwind CSS and custom design tokens. UI components leverage `shadcn/ui` (based on Radix UI primitives) adopting a "New York" aesthetic inspired by Linear's B2B design.

### Technical Implementations
- **Frontend**: React with TypeScript, Vite, Wouter, and TanStack Query.
- **Backend**: Node.js with Express and TypeScript (ESM modules).
- **Data Storage**: PostgreSQL (Neon serverless) with Drizzle ORM.
- **Lead Generation Logic**: Employs conditional step execution, automatic data source fallback, and leverages historical performance data within a real-time, concurrent execution pipeline.
- **Chat Integration**: Uses a queue-based architecture with shared Supabase tables for AI interaction and intent-based routing.
- **Job Execution**: Manages background tasks such as nightly maintenance, syncing, monitoring, and lead generation with lifecycle management.
- **Agentic Decision Loop**: Integrates with the Tower Judgement API for plan evaluation (`CONTINUE`, `RETRY`, `CHANGE_PLAN`, `STOP`).
- **Logging**: A three-tier logging system provides comprehensive monitoring with structured logs.
- **Progress Tracking**: Uses an in-memory store for real-time tracking of concurrent plan executions.
- **Event System**: A Map-based registry per `planId` ensures isolated event streams during concurrent executions.
- **Deep Research**: A multi-provider research system generates reports based on user queries.
- **Intent Classification**: Routes messages with lead-finding verbs to `SEARCH_PLACES` and explicit deep research keywords to `DEEP_RESEARCH`.
- **Plan Execution**: Calls Tower via `judgeArtefact` after every step for step-level judgements. The final run verdict comes from a SINGLE authoritative Tower call on a `final_delivery` artefact, issued AFTER CVL verification completes. No Tower calls are made on `leads_list` or `attribute_verification` artefacts. The replan loop uses local count-based heuristics (`buildShortfallDirective`) instead of per-replan Tower calls. Invariant: `towerJudgedBeforeVerification = false` — Tower never judges leads before verification. If Tower returns STOP/FAIL on final_delivery, the verdict cannot flip to PASS.
- **Two-Phase Multi-Tool Execution**: Discovers leads with `SEARCH_PLACES` then enriches them using `WEB_VISIT`, `CONTACT_EXTRACT`, and `LEAD_ENRICH`, with `WEB_SEARCH` as a fallback.
- **Attribute Evidence Investigation Loop**: When HAS_ATTRIBUTE constraints require verification, the system uses a three-tier investigation strategy instead of only checking cached enrichment data. (1) `cached_pages`: uses WEB_VISIT pages from the enrichment phase; (2) `active_web_visit`: if a lead has a website but no cached pages (wasn't in the enrichment batch), actively calls WEB_VISIT to fetch and scan the site; (3) `web_search_then_visit`: if a lead has no website from Places, calls WEB_SEARCH to find a URL, then WEB_VISIT to scan it. Budget limits: `ATTR_MAX_ACTIVE_VISITS` (default 8) caps active WEB_VISIT calls, `ATTR_MAX_SEARCH_FALLBACKS` (default 5) caps WEB_SEARCH fallback calls. Each `attribute_evidence` artefact carries `investigation_strategy` and the summary `attribute_verification` artefact includes `investigation_breakdown` counts. A reusable `scanPagesForAttribute()` helper handles keyword matching identically across all three paths. `activeWebVisit()` and `webSearchForAttribute()` helper functions handle the tool calls with proper artefact creation and error handling.
- **RESTful API**: Manages leads, user context, signals, and plan execution, including creation, approval, and progress monitoring.
- **Constraint Verification Layer (CVL)**: An additive layer that extracts, checks, and performs per-lead deterministic verification, influencing the final verdict. `CATEGORY_EQUALS` constraints are **disabled** — business type is used as a search query term only, never as a verifiable constraint. The `PLACES_SUPPORTED_CATEGORIES` whitelist in `cvl.ts` is retained as dead code but never triggered because `c_category` constraints are filtered out at parse time and no longer safety-injected by the supervisor. This prevents niche business types (e.g. "micropubs") from being classified as `blocking_hard` and triggering CVL override stops. For `HAS_ATTRIBUTE` constraints, the CVL receives an `AttributeEvidenceMap` (keyed by placeId → attribute → {verdict, confidence, reason, evidenceUrl}) from the attribute verification gate, so per-lead attribute verdicts (yes/no/unknown) are incorporated into `verified_exact` counts and the CVL override logic. `LOCATION_EQUALS` uses geo bounding box verification via `server/supervisor/geo-regions.ts` — leads carry `lat`/`lng` from Google Places, and verification checks whether the point falls inside known region bounding boxes. Composite regions (e.g. "Sussex" = East Sussex + West Sussex) are supported. CVL uses a truth-preserving four-status model: `VerificationStatus = 'yes' | 'no' | 'unknown' | 'search_bounded'`. For SOFT location constraints, `search_bounded` counts toward `verified_exact` (the search was bounded to the region); for HARD constraints, only `yes` (VERIFIED_GEO) satisfies. Each lead carries a `location_confidence: LocationConfidence` field (`verified_geo`, `search_bounded`, `out_of_area`, `unknown`, `not_applicable`). Summary includes `location_breakdown: { verified_geo_count, search_bounded_count, out_of_area_count, unknown_count }` and per-constraint `leads_search_bounded` counts.
- **Intelligent Replanning**: Separates user-requested counts from search budgets, accumulates and deduplicates leads across replan versions, employs a progressive geographic expansion strategy, enforces hard constraints, and includes early stopping. When no user count is specified, the system performs a single Google page 1 search (budget=20), skips count-based replanning, and treats all returned results as complete (no shortfall, no "missing" messaging). Count-based replan overrides, Learning Layer budget caps, and `requested_count` hard constraints only activate when the user explicitly specifies a number. The supervisor separates `searchBudgetCountFromGoal` (internal budget, 30–50 for search) from `userRequestedCountFinal` (user-facing cap). All user-facing artefacts (`leads_list`, `final_delivery`, `dsLeads` for delivery summary) are capped to `userRequestedCountFinal` before emission. A `count_diagnostic` artefact is emitted after every `leads_list` artefact, logging `userRequestedCountFinal`, `searchBudgetCount`, `candidateCountFromGoogle`, and `deliveredCountAfterTrim` for pipeline transparency.
- **Canonical Requested Count**: All runs carry a `RequestedCountCanonical` object (`requested_count_user: "explicit" | "any"`, `requested_count_value: number | null`, `requested_count_effective: number`). `DEFAULT_LEADS_TARGET = 20`. When user specifies no count → `{user: "any", value: null, effective: 20}`. Every Tower `successCriteria` includes all three fields plus `target_count = requested_count_effective` (never null). This eliminates Tower STOPs caused by missing count fields. Defined in `goal-to-constraints.ts` via `buildRequestedCount()`. Regression tests in `requested-count.test.ts`.
- **LLM-backed Goal-to-Constraints Parser**: Converts natural language user goals into structured constraints using LLMs and strict JSON schema validation. Country is inferred from location (US states → "US", UK locations → "UK") rather than defaulting to "UK". The `inferCountryFromLocation` function provides regex fallback country detection. `sanitiseLocationString()` strips instruction contamination from location fields using `LOCATION_INSTRUCTION_PATTERNS` (a shared pattern array also used by `stripInstructionClauses`) covering "and include email/phone/website", "including email", "with email", parenthesised forms, and comma/semicolon variants. A keyword-based fallback guard (`INSTRUCTION_KEYWORD_RE`) catches residual contamination by splitting on commas to extract the city-only prefix. In the supervisor, `searchQuery.location` is only used as a fallback when `parsedGoal.location` is empty, and is always sanitised first — preventing LLM-polluted overrides (e.g. "arundel and include email"). `detectExactnessMode()` detects hard vs soft exactness intent. `detectDoNotStop()` detects "do not stop" phrases (logged, ignored — budgets enforced instead). `stripInstructionClauses()` removes instruction text from raw goals before constraint extraction. `detectDeliveryRequirements()` detects "include email/phone/website" as delivery requirements, never contaminating location. `ParsedGoal` includes `include_email`, `include_phone`, `include_website` boolean fields. Contact requests (email/phone/website) are tracked as `contactRequests` in the supervisor and passed to artefacts as `contact_requests` metadata with explicit notes that they are enrichment-only, NOT search or pass/fail constraints. The plan artefact's `original_user_goal` uses `normalizedGoal` (sanitised) instead of `originalUserGoal` (raw) to prevent Tower from interpreting delivery requirements as hard constraints.
- **Evidence Extraction Step**: After WEB_VISIT for attribute queries, the `extractEvidenceSnippets()` function in `server/supervisor.ts` performs sentence-level evidence extraction from `text_clean`. Text is split into sentences (by `.!?` boundaries), each scored by keyword relevance (match count, position, sentence length), and the top 1-3 snippets are stored. Fallback to 200-char chunk extraction if no sentences detected. The `attribute_evidence` artefact now includes: `extracted_quotes: string[]` (sentence-level snippets from page body, not just page titles), `page_url`, `page_title`, `extraction_method` (`keyword_sentence_match` | `keyword_window` | `no_match` | `none`), `original_goal`, `constraint_raw`. The `evidence.quote` field is updated to prefer the best extracted sentence over the old narrow keyword-window snippet. `run-receipt.ts` prefers `extracted_quotes[0]` over `evidence.quote` for snippet display. `PageScanResult` interface extended with `extractedQuotes`, `pageUrl`, `pageTitle`, `extractionMethod`. Log line includes `extractedQuotes` count and `method`. No changes to Tower judgement logic or verdict scoring.
- **Run Receipt Artefact**: `server/supervisor/run-receipt.ts` emits a `run_receipt` artefact after every completed run. This is the canonical, user-facing "what happened" summary that the UI can render directly. The receipt matches `lead_pack` and `contact_extract` artefacts to **delivered leads only** (by place_id, then normalised name, then title substring) and computes proven contact counts. CONTACT_EXTRACT artefacts are discovered from two sources: legacy `type: 'contact_extract'` artefacts AND `type: 'step_result'` artefacts where `payload.step_type === 'CONTACT_EXTRACT'` (the production format). Contacts are extracted from `payload.outputs.contacts`, `payload.contacts`, or `payload.contact_extract_outputs.contacts`. Step-result CONTACT_EXTRACT artefacts carry `lead_place_id` and `lead_name` in their payload for robust matching. `contacts_proven` is `false` if no contact artefacts could be matched — in that case, counts are `null` and the narrative uses safe language ("Contact details varied by venue"). **Attribute outcomes**: `run_receipt.outcomes.attributes` (optional, present when `attribute_evidence` or `attribute_verification` artefacts exist) summarises HAS_ATTRIBUTE verification results per attribute. Each entry has `attribute_raw`, `matched_count`, `matched_place_ids`, `unknown_count`, and `evidence_refs` (with `place_id`, `url`, optional `snippet` and `matched_variant`). Only `verdict=yes` with a non-empty `evidence.source_url` counts as matched; everything else is unknown. Primary source: `attribute_evidence` artefacts; fallback: `attribute_verification` artefact's `results[]` array. Narrative lines include attribute verification summaries. The `narrative_lines` array contains non-technical chat-bubble text. A `debug` object includes `artefact_ids_used` (with `lead_pack`, `contact_extract`, and `attribute_evidence` arrays), `counting_method`, and `notes`. Emitted after `emitDeliverySummary` and before goal status update. Tests in `run-receipt.test.ts` (15 tests).
- **Search Query Compiled Artefact**: `server/supervisor/search-query-compiled.ts` emits a `search_query_compiled` artefact at end of every run, capturing `interpreted_location`, `interpreted_query`, `requested_count`, `exactness_mode`, `do_not_stop_ignored`, radius escalation stats, candidate counts from Google, `stop_reason`, `query_broadening_applied`, and `query_broadening_terms` for full pipeline transparency.
- **Synonym Broadening (Underfill Recovery)**: `QUERY_SYNONYM_MAP` and `getQueryBroadeningTerms()` in `replan-policy.ts` provide synonym expansion (e.g. "pubs" → "pubs OR bar OR inn") as a last-resort underfill recovery when radius expansion is exhausted and no progress is being made. Broadening is tracked via `queryBroadeningApplied`/`queryBroadeningTerms` in the supervisor and surfaced in the `search_query_compiled` artefact.
- **Canonical Delivery Summary**: `delivery_summary` provides a canonical `status` field (`COMPLETED`|`STOP`|`ERROR`) and `trust_status` field (`TRUSTED`|`UNTRUSTED`) derived from delivered lead count, serving as the authoritative user-facing output. `deriveCanonicalStatus()` uses a simple rule: if leads were delivered (`totalDelivered > 0`), status is `COMPLETED` regardless of Tower verdict; if no leads and Tower error, status is `ERROR`; otherwise `STOP`. The Tower verdict (`pass`/`fail`/`stop`) is preserved in `tower_verdict` field as metadata but does NOT determine the delivery summary status. Terminal state in `agent_runs` is always `completed` (never `stopped`) when the pipeline reaches final delivery. `final_delivery` payload includes `requested_count`, `delivered_count`, and `found_count_raw`; leads are trimmed to `requested_count` before final delivery to prevent over-delivery. `TRUSTED` = COMPLETED; `UNTRUSTED` = STOP or ERROR. The `cvl_summary` field includes `location_breakdown: { verified_geo_count, search_bounded_count, out_of_area_count, unknown_count }` for honest UI bucketing. The `leads` artefact payload attaches per-lead `verification: { location_status, location_confidence, verified_exact, all_hard_satisfied, verification_level }` so the UI can render verification status without inference. `verification_level` is `'verified'` (CVL checks + exact), `'checked'` (CVL checks + not exact), `'candidate'` (CVL match but 0 checks), or `'unverified'` (no CVL match). `verified_exact` is never `true` unless CVL actually ran checks (vacuous-truth guard in `verifyLeads`). Classification into `delivered_exact[]` vs `delivered_closest[]` uses CVL per-lead verification for location constraints when available, falling back to substring heuristics only when CVL data is absent. `delivered_exact_count` always equals `delivered_exact.length` — no count/array mismatch. Final message metadata includes `final_verdict` (canonical status) and `trust_status` for downstream consumers.
- **Canonical Intent Wiring (Semantic Overreach Reduction)**: When the intent extractor produces a valid `CanonicalIntent` (shadow mode active), downstream gates trust its structured fields instead of re-parsing raw user messages with regex. `evaluateClarifyGateFromIntent()` uses `mission_type` for routing; `preExecutionConstraintGateFromIntent()` builds constraint contracts from `intent.constraints[]`; `classifyFollowUp()` accepts optional `followUpIntent` for scope-change detection; `evaluatePreflightClarify()` uses canonical `entity_category`/`location_text`/time constraints. All paths fall back to legacy regex when canonical intent is unavailable or `mission_type=unknown`. Every gate result carries a `semantic_source: 'canonical' | 'fallback_regex'` observability marker logged in artefacts and console. Tower judgement logic is unchanged. **Semantic overreach reduction**: When canonical intent is active: (1) regex field extractors (`extractBusinessType`, `extractLocation`, `extractCount`, `extractTimeFilter`) are skipped entirely — canonical fields are the sole truth source; (2) relationship predicate detection trusts canonical constraint types — if canonical has no `relationship` constraint, `detectRelationshipPredicate()` is NOT called on raw text, preventing false positives like "serve vegan food" being misclassified as a relationship; (3) `detectExactnessMode()` and `detectDoNotStop()` are derived from canonical constraint data instead of regex on raw goals; (4) time predicate regex fallback in `evaluatePreflightClarify()` is skipped when canonical intent is present; (5) `preExecutionConstraintGateFromIntent()` trusts canonical `evidence_mode=website_text` to auto-resolve live_music verification choices without re-scanning raw text. Visibility fields: `intent_source: 'canonical' | 'legacy'` on `plan_update` and `final_delivery` artefacts; `evidence_ready_for_tower: boolean` on `attribute_verification` and `final_delivery` artefacts.
- **Clarify Gate**: A strict intent-routing layer in `server/supervisor/clarify-gate.ts` that runs BEFORE any agent execution in `processChatTask()`. Routes every message to exactly one of three routes: `direct_response` (explanations, meta questions, trust queries — no tools), `clarify_before_run` (asks 1–3 targeted questions — no tools), or `agent_run` (proceeds to execution). Meta/trust questions (e.g. "can I trust", "how accurate", "guaranteed") are detected by `isMetaTrust()` and always route to `direct_response` even if the message contains search-like patterns — this check runs BEFORE `hasSearchIntent`. **Tightened gate criteria (2026-03-04):** `clarify_before_run` now ONLY triggers for three narrow categories: (1) **empty input** — empty or whitespace-only messages; (2) **nonsense/unintelligible input** — gibberish strings with very few recognisable English words (detected by `isNonsenseInput()`); (3) **multiple concatenated requests** — detected by `isMalformedInput()` (camelCase sentence boundaries, multiple search verbs) or `hasMixedIntent()` ("and also find", "plus find", etc.). All other single requests — including those with subjective terms ("best", "nice", "good"), missing locations, vague entity types, relationship predicates, vague proximity, or false prior context — now proceed to `agent_run` and are handled by downstream systems. A `triggerCategory` diagnostic field (`'empty' | 'multiple_requests' | 'malformed' | 'unknown'`) is included in every `clarify_before_run` result. The `ClarifyGateResult` interface includes `triggerCategory?: ClarifyTriggerCategory`. The supervisor emits `triggerCategory` in the diagnostic artefact payload and console log. Tests in `clarify-gate.test.ts`.
  - Examples: `"find the best cafes in Bristol"` → `agent_run`; `"find cafes in Bristol"` → `agent_run`; `"find pubs in LeedsShow me cafes"` → `clarify_before_run` (triggerCategory=multiple_requests); `""` → `clarify_before_run` (triggerCategory=empty); `"asdfgh jklzxcv"` → `clarify_before_run` (triggerCategory=malformed).
- **Clarify Session**: `server/supervisor/clarify-session.ts` provides in-memory per-conversation session state for the clarify flow. Tracks `originalUserRequest` (immutable), `originRunId` (the `agent_run.id` from the first message that opened the session — all follow-up messages reuse this run_id for artefact continuity in AFR), `missingFields`, and `collectedFields` (structured, never concatenated text). When a follow-up message arrives and the session has an `originRunId`, `processChatTask()` overrides `jobId` to the `originRunId` so all artefacts (constraints, diagnostics, follow-ups) accumulate on a single run in AFR. The origin run uses `status: 'clarifying'` with `terminalState: null` and `awaiting: 'user_input'` in metadata — it is resumed to `executing` when the follow-up task creates an agent_run with the same ID (duplicate key → update). `collectedFields` includes `businessType`, `location`, `attributes` (string array — refinements accumulate, not replace), `count`, `relationship`, `timeFilter`, and `semanticConstraint` (stores the subjective intent phrase, e.g. "best vibes", until clarified into measurable attributes — NOT treated as runnable). Follow-up messages are classified as `ANSWER_TO_MISSING_FIELD`, `REFINEMENT`, `NEW_REQUEST`, `META_TRUST`, or `EXECUTE_NOW`. `NEW_REQUEST` immediately closes the session and routes the message normally. **Scope-change escape (G4)**: `isContradictoryNewTask(msg, session)` runs before other classification — detects when a follow-up introduces a different entity type (e.g. "cafes" during a "pubs" session) with its own location or a search verb, and returns `NEW_REQUEST` to exit the session and create a fresh task. Prefers false negatives over false positives; same-entity messages ("pubs in Brighton" during a pubs session) are NOT treated as contradictions. `META_TRUST` answers the trust/meta question directly while preserving the session. `EXECUTE_NOW` recognizes explicit execute commands ("search now", "run it", "go ahead", "yes proceed", "do it", etc.) — if the draft is complete, the search runs immediately; if incomplete, the system tells the user exactly what fields are missing rather than absorbing execute-intent text as field values. Sessions are bounded to `MAX_CLARIFY_TURNS` (3) — after the cap, the user is offered choices: run as-is, start over, or ask something else. Session auto-expires after 15 minutes. Every clarify response includes a structured `clarify_state` in message metadata: `{ status: 'ask_more' | 'ready_to_search' | 'turn_limit_reached', draft: {...}, missingFields: [...], turnCount, maxTurns }`. Follow-up classification uses `MEASURABLE_CRITERIA` (synced with gate's `MEASURABLE_ATTRIBUTES`) to recognise measurable terms as answers to `semantic_constraint` — covers nightlife, lively, romantic, trendy, walkable, events, student, views, scenic, good for studying, cosy, quiet, late night, and all `BUSINESS_MODIFIERS`. Tests in `clarify-session.test.ts` (77 tests).
- **Time Predicate Contracts**: `server/supervisor/time-predicate.ts` implements the Constraint Truth Layer for time-based predicates (e.g. "opened recently", "opened in last 12 months", "newly opened", "just opened", "closed recently", "renovated"). Detects time predicate phrases via `detectTimePredicate()`, normalises window to days, infers `hardness` (hard when "must have"/"definitely" present), and builds a `TimePredicateContract` with `verifiability='proxy'` (opening dates cannot be verified from Google Places). Supported proxy options: `recent_reviews` (review spike/first review within window) and `news_mention` (web evidence of opening). Unsupported: `new_listing` (no first-seen index), `companies_house_incorp` (companies only, not available). `resolveProxyChoice()` handles user proxy selection: accepted supported proxy → `can_execute=true`; rejected ("no proxies") → `hard+unverifiable`, `can_execute=false` (STOP); unsupported proxy → `can_execute=false`. `buildClarifyQuestion()` generates a tight proxy-selection question. `buildHonestyLine()` generates delivery summary honesty text when proxies were used. Does NOT modify G1–G7 decision tree routing. Tests in `time-predicate.test.ts` (48 tests). Authority comment at top of file: "Supervisor is the sole authority for constraint interpretation and verifiability."
- **Pre-Execution Constraint Gate**: `server/supervisor/constraint-gate.ts` implements the hard "Pre-Execution Constraint Gate" that runs BEFORE any tool execution, Google search, or SEARCH_PLACES call. Extracts ALL constraints from a single message: subjective predicates, numeric ambiguity, relationship predicates (via `relationship-predicate.ts`), time predicates (via `time-predicate.ts`), and blocking attribute constraints (`live_music` requires clarification unless the initial message contains explicit website-verification phrasing like "on their website", "from their website", "website says/mentions", in which case the constraint is auto-resolved to `website_verify` without asking; other attributes like `beer_garden`, `craft_beer` pass through as non-blocking). `preExecutionConstraintGate(msg)` returns a `ConstraintContract` with `constraints[]`, `can_execute`, `why_blocked`, `clarify_questions[]`, `stop_recommended`. NON-NEGOTIABLE RULE: if any constraint is unresolved, `can_execute=false` — no search bubble, no Google query, no tools. **Clarification messages use specific wording**: time predicate asks ONE question with 3 options (A: news mentions proxy, B: first reviews proxy, C: best-effort unverified); live music asks ONE question with 2 options (A: verify via website/listings, B: best-effort unverified); relationship predicates ask ONE question with 4 options (A: official sources only, B: best-effort public web, C: 2+ independent sources, D: skip relationship fields if uncertain). **Sequential clarification**: exactly one active blocking constraint drives the UI at a time. Priority order: subjective → numeric ambiguity → relationship predicate → time predicate → live music/attributes. After each constraint is resolved, the next unresolved one surfaces. Resolved constraints are never re-surfaced and cannot be overwritten by signals intended for later constraints. **Relationship Predicate Gating (Batch 4)**: `RelationshipPredicateConstraint` type with `verifiability='proxy'`, `hardness='soft'`, `can_execute=false` until user selects an approach. Detects relationship predicates (owned by, run by, managed by, operated by, part of) and relationship roles (owner, landlord, manager, operator, contact person, decision maker, GM, general manager, head brewer, practice manager, freehouse, free house, tied house, group, chain) via `RELATIONSHIP_ROLE_PATTERNS` in `relationship-predicate.ts`. Resolution stores `chosen_relationship_strategy` (official_only, best_effort_web, two_plus_sources, skip_if_uncertain) and sets `can_execute=true`. Honesty: if strategy is best_effort_web, relationship fields marked as unverified in outputs metadata; if official_only or two_plus_sources, marked verified only when evidence meets criteria. Tests in `constraint-gate.test.ts` (22 relationship-specific tests). **Numeric Ambiguity Gating (Batch 3)**: `NumericAmbiguityConstraint` type with three categories: `fuzzy_quantity` ("a few", "many", "several" — undefined count), `ranking` (bare "top"/"best" without count — undefined ranking criteria/count), `numeric_adjective` ("cheap", "expensive", "large" — undefined threshold). "top 5"/"best 10" pass through (count provided). "best rated"/"best places" remain subjective. Resolution is category-specific: fuzzy_quantity requires a count (`\b\d+\b`), ranking requires a count, numeric_adjective requires a threshold with currency/units. `resolveFollowUp(contract, followUpMsg)` handles multi-turn resolution for both time predicate (proxy/best-effort/no-proxy) and live music (website-verify/best-effort). `detectNoProxySignal(msg)` detects "no proxies"/"must be certain"/"don't guess" → HARD+unverifiable, STOP. `detectMustBeCertain(msg)` detects "must be certain"/"must be guaranteed"/"must be verified"/"need to be certain"/"require certainty"/"has to be certain"/"i need certainty"/"certainty is required"/"only if you're certain" → applies `applyCertaintyGate()` which sets `must_be_certain=true` on all constraints, then checks each via `isCertainVerifiable()`: time_predicate → always unverifiable; live_music → always unverifiable; verifiable attributes (beer_garden, etc.) → pass through. Unverifiable constraints with `must_be_certain=true` → `can_execute=false`, `stop_recommended=true`, STOP with alternatives message. Works in both initial message (`preExecutionConstraintGate`) and follow-up (`resolveFollowUp`). `must_be_certain` boolean added to both `TimePredicateContract` and `AttributeConstraint` interfaces. Outer gate in `supervisor.ts` also checks original request for must-be-certain signals via `detectMustBeCertain()`. `detectBestEffort(msg)` detects "best effort"/"unverified is ok"/"option C" → accepts unverified. `detectLiveMusicChoice(msg)` detects "verify via website"/"A)"/"B)" for live music. Pending contracts stored per-conversation with TTL. Wired into `supervisor.ts` at THREE points: (1) **OUTER GATE in `processChatTask()`** — runs BEFORE `executeTowerLoopChat()` is called, preventing agent_run creation, LLM calls, or any observable side-effects when a time predicate exists; (2) `processChatTask()` handles follow-up messages for pending constraint contracts (restores original message on resolution); (3) `executeTowerLoopChat()` runs the gate as defense-in-depth right before SEARCH_PLACES plan is built (skips if already resolved via `_constraint_gate_resolved` flag). **Best-effort acceptance sets `hardness='soft'`** to prevent false STOP when original message contained hard language like "must have". Tests in `constraint-gate.test.ts` (61 tests). Total test count: 231 across 4 test files.
- **Pre-Plan Gate**: Before plan execution, `evaluatePrePlanGate()` checks for vertical mismatches (e.g. "social care" in a brewery vertical), informational queries (questions without lead-finding verbs), and merged queries (multiple search intents in one message). Vertical mismatch and informational queries trigger `clarification_needed=true` and emit a `clarification_needed` artefact, returning early without calling SEARCH_PLACES. Merged queries are flagged as `query_suspected_merged=true` (warning-only, does not block). The gate is wired into `executeTowerLoopChat()` and the decision log includes the merged-query flag.
- **Session Isolation Guard**: Module-level `_activeTaskPerConversation` map tracks `conversationId → { taskId, runId }`. `registerActiveTask()` called at the start of `processChatTask()` registers the current task as active. `guardDelivery(convId, taskId, label)` and `isRunCurrentForConversation(convId, runId)` check staleness before any output delivery. Guards placed at: (1) `postArtefactToUI` — blocks stale artefact POSTs (also now includes `conversationId` in the POST body); (2) final message delivery after `executeTowerLoopChat` returns; (3) after SEARCH_PLACES completes (before enrichment); (4) before enrichment loop; (5) at top of replan loop iteration. ### System Design Choices
- **IStorage Interface**: Provides an abstraction for database operations.
- **Logging Infrastructure**: Robust, structured logging integrated with Tower.
- **Concurrent Execution Handling**: In-memory progress tracking and a Map-based event system ensure isolated and efficient handling.
- **Error Handling**: Comprehensive error handling with proper status updates for failed plans.
- **Completion Gating**: Ensures `run_completed` is emitted only with Tower approval for `SEARCH_PLACES` runs.
- **Tower AFR Provability**: Every `SEARCH_PLACES` Tower call emits specific AFR events and a `tower_judgement` artefact.
- **Live Activity `clientRequestId` Threading**: `RunState` carries `clientRequestId` for correlation across all Tower events.
- **Mandatory Inline Tower Observation**: Always-on inline Tower observation with fatal failures.
- **Plan Executor Per-Step Tower Judgement**: The plan executor calls Tower via `judgeArtefact` after every step, enabling a bounded retry/replan inner loop.
- **Bypass Detector**: Detects runs that bypass the Supervisor.
- **Short-Circuit Diagnostics**: Emits `diagnostic` artefacts and AFR events when runs short-circuit due to concurrency guard, execution errors, or unhandled exceptions, ensuring at least one artefact is always visible for any agent run.
- **Canonical Run ID Propagation**: `startJob` for `deep_research` sets `run_id` as a column AND in `request_data`, using the caller-provided `run_id` or `jobId` as the canonical ID, preventing run ID mismatches between the routing layer and supervisor.
- **Batch-Claim + Background Claimer**: The supervisor atomically claims ALL pending tasks upfront before processing any, preventing the external Wyshbone UI from racing to grab tasks during long-running executions. A background claimer runs every 2 seconds (via `setInterval`) to claim tasks that arrive while the main loop is busy processing, queuing them for immediate processing after the current batch.
- **Orphaned Agent Runs Recovery**: On startup, `recoverOrphanedAgentRuns()` finds `agent_runs` stuck in `executing` state for > 3 minutes and marks them as `failed` with `orphan_reason: 'server_restart'` in metadata. This handles runs that were interrupted by server restarts where the `supervisor_tasks` recovery couldn't help (task already completed/failed but agent_run left in executing). The `sweepStaleTasks` and `recoverOrphanedTasks` both now check for `claimed` status in addition to `processing`.
- **Canonical Intent Extractor (Active Primary Parser)**: `server/supervisor/canonical-intent.ts` defines the `CanonicalIntent` Zod schema — a single source of truth for extracted user intent with enum-enforced fields: `mission_type`, `entity_kind`, `entity_category`, `location_text`, `geo_mode`, `radius_km`, `requested_count`, `default_count_policy`, `constraints[]` (each with `type`, `raw`, `hardness`, `evidence_mode`, `clarify_if_needed`, `clarify_question`), `plan_template_hint`, `preferred_evidence_order`. `server/supervisor/intent-extractor.ts` provides `extractCanonicalIntent(msg, conversationContext?)` — calls OpenAI gpt-4o-mini or Anthropic claude-3-5-haiku with an upgraded strict JSON-only prompt containing 4 few-shot examples, detailed field rules, negative rules, and conversation context handling. `server/supervisor/intent-shadow.ts` implements the feature flag `INTENT_EXTRACTOR_MODE` (off|shadow|active|strict). **The canonical extractor is now the ACTIVE primary parser**: when extraction is valid, `canonicalIntentToParsedGoal()` builds a full `ParsedGoal` from `CanonicalIntent` (mapping entity_category→business_type, location_text→location, canonical constraints→StructuredConstraint[], geo_mode→LOCATION_NEAR/LOCATION_EQUALS, inferring country from location). Validation rejects old v1 fields (`action`, `business_type`, `country`, `location`, `count`, `delivery_requirements`, `confidence`, `raw_input`) with explicit error message. `server/supervisor/intent-extractor.ts` calls gpt-4o-mini (fallback claude-3-5-haiku) with a JSON-only prompt to extract structured intent. `server/supervisor/intent-shadow.ts` provides `runIntentExtractorShadow()` which runs in shadow mode (observes without changing planner behaviour) when `INTENT_EXTRACTOR_MODE=shadow`. Emits two artefacts per run: `intent_extracted_shadow` (extraction result including validation status, always emitted even on LLM failure) and `diagnostic` (mode, executed, validation_passed). The call is `await`-ed (not fire-and-forget) in `processChatTask()` to guarantee artefact persistence. 50 unit tests in `canonical-intent.test.ts`.

### UI inFlight Spinner Fix
- **Root cause**: `task_execution_started` AFR event was emitted at the top of `processChatTask()` but NO matching `task_execution_completed` event was emitted on clarify/early-return paths. The UI tracks `inFlight` using paired events — without a matching completion event, the counter stays at 1 and the spinner never settles.
- **Fix**: `emitTaskExecutionCompleted(verdict, extraMeta?)` helper function defined before the main try block in `processChatTask()`, guarded by `taskExecutionCompleteEmitted` (prevents double-emit) and `taskExecutionStartedEmitted` (prevents emission if started event never fired). Wired to ALL 15 exit paths: `preflight_clarify`, `factory_demo_stale`, `factory_demo`, `constraint_gate_stop`, `constraint_gate_clarify`, `meta_trust_during_session`, `execute_blocked_incomplete`, `clarify_turn_limit`, `clarify_session_continue`, `direct_response`, `clarify_before_run`, `outer_constraint_gate`, `session_guard_stale_run`, `run_completed`, and `run_error` (catch block).

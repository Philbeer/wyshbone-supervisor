# Wyshbone Supervisor Suite

## Overview
Wyshbone Supervisor is a B2B lead generation system designed for automatic prospect identification and scoring. It delivers real-time lead suggestions with contact information via email and an integrated chat. The system aims to enhance sales processes, expand market reach, provide actionable data, and improve workflow efficiency. Future ambitions include integrating AI for plan evaluation, deep research, and intelligent replanning to optimize lead generation and verification.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### UI/UX Decisions
The frontend uses React, TypeScript, Vite, and Wouter. Styling is managed with Tailwind CSS and custom design tokens. UI components leverage `shadcn/ui` (based on Radix UI primitives) adopting a "New York" aesthetic inspired by Linear's B2B design.

### Technical Implementations
- **Frontend**: React with TypeScript, Vite, Wouter, and TanStack Query.
- **Backend**: Node.js with Express and TypeScript (ESM modules).
- **Data Storage**: PostgreSQL (Neon serverless) with Drizzle ORM.
- **Lead Generation Logic**: Employs conditional step execution, automatic data source fallback, and leverages historical performance data within a real-time, concurrent execution pipeline.
- **Chat Integration**: Uses a queue-based architecture with shared Supabase tables for AI interaction and intent-based routing.
- **Job Execution**: Manages background tasks such as nightly maintenance, syncing, monitoring, and lead generation with lifecycle management.
- **Agentic Decision Loop**: Integrates with the Tower Judgement API for plan evaluation (`CONTINUE`, `RETRY`, `CHANGE_PLAN`, `STOP`).
- **Logging**: A three-tier logging system provides comprehensive monitoring with structured logs.
- **Progress Tracking**: Uses an in-memory store for real-time tracking of concurrent plan executions.
- **Event System**: A Map-based registry per `planId` ensures isolated event streams during concurrent executions.
- **Deep Research**: A multi-provider research system generates reports based on user queries.
- **Intent Classification**: Routes messages with lead-finding verbs to `SEARCH_PLACES` and explicit deep research keywords to `DEEP_RESEARCH`.
- **Plan Execution**: Calls Tower via `judgeArtefact` after every step, with a bounded retry/replan inner loop.
- **Two-Phase Multi-Tool Execution**: Discovers leads with `SEARCH_PLACES` then enriches them using `WEB_VISIT`, `CONTACT_EXTRACT`, and `LEAD_ENRICH`, with `WEB_SEARCH` as a fallback.
- **RESTful API**: Manages leads, user context, signals, and plan execution, including creation, approval, and progress monitoring.
- **Constraint Verification Layer (CVL)**: An additive layer that extracts, checks, and performs per-lead deterministic verification, influencing the final verdict. `CATEGORY_EQUALS` constraints are **disabled** — business type is used as a search query term only, never as a verifiable constraint. The `PLACES_SUPPORTED_CATEGORIES` whitelist in `cvl.ts` is retained as dead code but never triggered because `c_category` constraints are filtered out at parse time and no longer safety-injected by the supervisor. This prevents niche business types (e.g. "micropubs") from being classified as `blocking_hard` and triggering CVL override stops. For `HAS_ATTRIBUTE` constraints, the CVL receives an `AttributeEvidenceMap` (keyed by placeId → attribute → {verdict, confidence, reason, evidenceUrl}) from the attribute verification gate, so per-lead attribute verdicts (yes/no/unknown) are incorporated into `verified_exact` counts and the CVL override logic. `LOCATION_EQUALS` uses geo bounding box verification via `server/supervisor/geo-regions.ts` — leads carry `lat`/`lng` from Google Places, and verification checks whether the point falls inside known region bounding boxes. Composite regions (e.g. "Sussex" = East Sussex + West Sussex) are supported. CVL uses a truth-preserving four-status model: `VerificationStatus = 'yes' | 'no' | 'unknown' | 'search_bounded'`. For SOFT location constraints, `search_bounded` counts toward `verified_exact` (the search was bounded to the region); for HARD constraints, only `yes` (VERIFIED_GEO) satisfies. Each lead carries a `location_confidence: LocationConfidence` field (`verified_geo`, `search_bounded`, `out_of_area`, `unknown`, `not_applicable`). Summary includes `location_breakdown: { verified_geo_count, search_bounded_count, out_of_area_count, unknown_count }` and per-constraint `leads_search_bounded` counts.
- **Intelligent Replanning**: Separates user-requested counts from search budgets, accumulates and deduplicates leads across replan versions, employs a progressive geographic expansion strategy, enforces hard constraints, and includes early stopping. When no user count is specified, the system performs a single Google page 1 search (budget=20), skips count-based replanning, and treats all returned results as complete (no shortfall, no "missing" messaging). Count-based replan overrides, Learning Layer budget caps, and `requested_count` hard constraints only activate when the user explicitly specifies a number.
- **Canonical Requested Count**: All runs carry a `RequestedCountCanonical` object (`requested_count_user: "explicit" | "any"`, `requested_count_value: number | null`, `requested_count_effective: number`). `DEFAULT_LEADS_TARGET = 20`. When user specifies no count → `{user: "any", value: null, effective: 20}`. Every Tower `successCriteria` includes all three fields plus `target_count = requested_count_effective` (never null). This eliminates Tower STOPs caused by missing count fields. Defined in `goal-to-constraints.ts` via `buildRequestedCount()`. Regression tests in `requested-count.test.ts`.
- **LLM-backed Goal-to-Constraints Parser**: Converts natural language user goals into structured constraints using LLMs and strict JSON schema validation. Country is inferred from location (US states → "US", UK locations → "UK") rather than defaulting to "UK". The `inferCountryFromLocation` function provides regex fallback country detection.
- **Canonical Delivery Summary**: `delivery_summary` provides a canonical `status` field (`PASS`|`PARTIAL`|`STOP`|`ERROR`) and `trust_status` field (`TRUSTED`|`UNTRUSTED`) derived from Tower verdict and CVL verified counts, serving as the authoritative user-facing output. `TRUSTED` = PASS or PARTIAL; `UNTRUSTED` = STOP or ERROR. The `cvl_summary` field includes `location_breakdown: { verified_geo_count, search_bounded_count, out_of_area_count, unknown_count }` for honest UI bucketing. The `leads` artefact payload attaches per-lead `verification: { location_status, location_confidence, verified_exact, all_hard_satisfied, verification_level }` so the UI can render verification status without inference. `verification_level` is `'verified'` (CVL checks + exact), `'checked'` (CVL checks + not exact), `'candidate'` (CVL match but 0 checks), or `'unverified'` (no CVL match). `verified_exact` is never `true` unless CVL actually ran checks (vacuous-truth guard in `verifyLeads`). Classification into `delivered_exact[]` vs `delivered_closest[]` uses CVL per-lead verification for location constraints when available, falling back to substring heuristics only when CVL data is absent. `delivered_exact_count` always equals `delivered_exact.length` — no count/array mismatch. Final message metadata includes `final_verdict` (canonical status) and `trust_status` for downstream consumers.
- **Clarify Gate**: A strict intent-routing layer in `server/supervisor/clarify-gate.ts` that runs BEFORE any agent execution in `processChatTask()`. Routes every message to exactly one of three routes: `direct_response` (explanations, meta questions, trust queries — no tools), `clarify_before_run` (asks 1–3 targeted questions when intent is unclear — no tools), or `agent_run` (proceeds to execution). Meta/trust questions (e.g. "can I trust", "how accurate", "guaranteed") are detected by `isMetaTrust()` and always route to `direct_response` even if the message contains search-like patterns — this check runs BEFORE `hasSearchIntent`. Triggers clarification for: missing/vague location, vague entity types without sector, relationship predicates, mixed/switched intent, malformed/concatenated input, false prior context references, **subjective/unmeasurable criteria** (best, top, coolest, vibes, nicest, etc. without a measurable attribute — asks user to pick concrete attributes; adds `semantic_constraint` to `missingFields` which blocks execution until resolved), and **nonsense/invalid locations** (placeholder nouns like "council things", "random stuff", "somewhere" — explicitly tells user the phrase isn't a place and asks for a real city/town/area; `parsedFields.location` is nulled to prevent storing nonsense). **Vague proximity with real location** detection (G6 fix): when a message contains a vague proximity reference ("near the council", "near the town hall") alongside a real location ("in Leeds"), the gate asks what the proximity means (distance? which building?) and confirms the real location, rather than generically asking for a location. **parsedFields sanitisation** (G6 fix): when `location` is in `missingFields`, `parsedFields.location` is set to `null` (prevents storing invalid locations like "council things"); when `businessType` contains subjective terms, they are stripped globally (e.g. "the best vibes" → `null`, "the best cafes" → "cafes"). Subjective criteria are bypassed when a measurable attribute is present (e.g. "best pubs with live music" runs normally). The gate also extracts `count` (safe numeric extraction avoiding temporal numbers like "12 months") and `timeFilter` (e.g. "last 12 months") into `parsedFields`. Every routing decision is logged to console (`[CLARIFY_GATE]`) and emitted as a `diagnostic` artefact. Agent execution CANNOT start unless route === `agent_run`. Tests in `clarify-gate.test.ts` (46 tests). Documentation in `docs/clarify-gate.md`.2 tests). Documentation in `docs/clarify-gate.md`.
  - G6 examples: `"find the best vibes near council things"` → clarify (asks for real location + what "vibes" means); `"find cafes in Bristol"` → agent_run (no false positive); `"find the best cafes in Bristol"` → clarify (asks what "best" means); `"find pubs near the council in Leeds"` → clarify (asks what "near the council" means + confirms Leeds).
- **Clarify Session**: `server/supervisor/clarify-session.ts` provides in-memory per-conversation session state for the clarify flow. Tracks `originalUserRequest` (immutable), `missingFields`, and `collectedFields` (structured, never concatenated text). `collectedFields` includes `businessType`, `location`, `attributes` (string array — refinements accumulate, not replace), `count`, `relationship`, `timeFilter`, and `semanticConstraint` (stores the subjective intent phrase, e.g. "best vibes", until clarified into measurable attributes — NOT treated as runnable). Follow-up messages are classified as `ANSWER_TO_MISSING_FIELD`, `REFINEMENT`, `NEW_REQUEST`, `META_TRUST`, or `EXECUTE_NOW`. `NEW_REQUEST` immediately closes the session and routes the message normally. **Scope-change escape (G4)**: `isContradictoryNewTask(msg, session)` runs before other classification — detects when a follow-up introduces a different entity type (e.g. "cafes" during a "pubs" session) with its own location or a search verb, and returns `NEW_REQUEST` to exit the session and create a fresh task. Prefers false negatives over false positives; same-entity messages ("pubs in Brighton" during a pubs session) are NOT treated as contradictions. `META_TRUST` answers the trust/meta question directly while preserving the session. `EXECUTE_NOW` recognizes explicit execute commands ("search now", "run it", "go ahead", "yes proceed", "do it", etc.) — if the draft is complete, the search runs immediately; if incomplete, the system tells the user exactly what fields are missing rather than absorbing execute-intent text as field values. Sessions are bounded to `MAX_CLARIFY_TURNS` (3) — after the cap, the user is offered choices: run as-is, start over, or ask something else. Session auto-expires after 15 minutes. Every clarify response includes a structured `clarify_state` in message metadata: `{ status: 'ask_more' | 'ready_to_search' | 'turn_limit_reached', draft: {...}, missingFields: [...], turnCount, maxTurns }`. Follow-up classification uses `MEASURABLE_CRITERIA` (synced with gate's `MEASURABLE_ATTRIBUTES`) to recognise measurable terms as answers to `semantic_constraint` — covers nightlife, lively, romantic, trendy, walkable, events, student, views, scenic, good for studying, cosy, quiet, late night, and all `BUSINESS_MODIFIERS`. Tests in `clarify-session.test.ts` (77 tests).
- **Time Predicate Contracts**: `server/supervisor/time-predicate.ts` implements the Constraint Truth Layer for time-based predicates (e.g. "opened recently", "opened in last 12 months", "newly opened", "just opened", "closed recently", "renovated"). Detects time predicate phrases via `detectTimePredicate()`, normalises window to days, infers `hardness` (hard when "must have"/"definitely" present), and builds a `TimePredicateContract` with `verifiability='proxy'` (opening dates cannot be verified from Google Places). Supported proxy options: `recent_reviews` (review spike/first review within window) and `news_mention` (web evidence of opening). Unsupported: `new_listing` (no first-seen index), `companies_house_incorp` (companies only, not available). `resolveProxyChoice()` handles user proxy selection: accepted supported proxy → `can_execute=true`; rejected ("no proxies") → `hard+unverifiable`, `can_execute=false` (STOP); unsupported proxy → `can_execute=false`. `buildClarifyQuestion()` generates a tight proxy-selection question. `buildHonestyLine()` generates delivery summary honesty text when proxies were used. Does NOT modify G1–G7 decision tree routing. Tests in `time-predicate.test.ts` (48 tests). Authority comment at top of file: "Supervisor is the sole authority for constraint interpretation and verifiability."
- **Pre-Execution Constraint Gate**: `server/supervisor/constraint-gate.ts` implements the hard "Pre-Execution Constraint Gate" that runs BEFORE any tool execution, Google search, or SEARCH_PLACES call. Extracts ALL constraints from a single message: time predicates (via `time-predicate.ts`) and blocking attribute constraints (`live_music` requires clarification; other attributes like `beer_garden`, `craft_beer` pass through as non-blocking). `preExecutionConstraintGate(msg)` returns a `ConstraintContract` with `constraints[]`, `can_execute`, `why_blocked`, `clarify_questions[]`, `stop_recommended`. NON-NEGOTIABLE RULE: if any constraint is unresolved, `can_execute=false` — no search bubble, no Google query, no tools. **Clarification messages use specific wording**: time predicate asks ONE question with 3 options (A: news mentions proxy, B: first reviews proxy, C: best-effort unverified); live music asks ONE question with 2 options (A: verify via website/listings, B: best-effort unverified). **Compound queries** (time + live music) show BOTH questions in ONE bubble as numbered bullets — execution blocked until BOTH answered. `resolveFollowUp(contract, followUpMsg)` handles multi-turn resolution for both time predicate (proxy/best-effort/no-proxy) and live music (website-verify/best-effort). `detectNoProxySignal(msg)` detects "no proxies"/"must be certain" → HARD+unverifiable, STOP. `detectBestEffort(msg)` detects "best effort"/"unverified is ok"/"option C" → accepts unverified. `detectLiveMusicChoice(msg)` detects "verify via website"/"A)"/"B)" for live music. Pending contracts stored per-conversation with TTL. Wired into `supervisor.ts` at THREE points: (1) **OUTER GATE in `processChatTask()`** — runs BEFORE `executeTowerLoopChat()` is called, preventing agent_run creation, LLM calls, or any observable side-effects when a time predicate exists; (2) `processChatTask()` handles follow-up messages for pending constraint contracts (restores original message on resolution); (3) `executeTowerLoopChat()` runs the gate as defense-in-depth right before SEARCH_PLACES plan is built (skips if already resolved via `_constraint_gate_resolved` flag). **Best-effort acceptance sets `hardness='soft'`** to prevent false STOP when original message contained hard language like "must have". Tests in `constraint-gate.test.ts` (61 tests). Total test count: 231 across 4 test files.
- **Pre-Plan Gate**: Before plan execution, `evaluatePrePlanGate()` checks for vertical mismatches (e.g. "social care" in a brewery vertical), informational queries (questions without lead-finding verbs), and merged queries (multiple search intents in one message). Vertical mismatch and informational queries trigger `clarification_needed=true` and emit a `clarification_needed` artefact, returning early without calling SEARCH_PLACES. Merged queries are flagged as `query_suspected_merged=true` (warning-only, does not block). The gate is wired into `executeTowerLoopChat()` and the decision log includes the merged-query flag.
- **Goal Ledger + Belief Store + Feedback Events**: Supabase tables track user goals, beliefs derived from failures, and user feedback actions.
- **Relationship Predicate Detection**: `server/supervisor/relationship-predicate.ts` detects relationship predicates ("works with", "supplies", "serves", "partners with", "clients of", etc.) in user intent and sets `requires_relationship_evidence=true`. When active: all Google Places results are candidates only (not matches); `verified_exact` is forced `false` per-lead; the delivery summary demotes all exact leads to closest with `relationship_not_verified` violation; `stop_reason` explains the relationship could not be verified; the chat response uses honest language ("found organisations associated with X, but could not verify…"); leads artefact summary says "candidates (relationship not verified)"; Tower successCriteria receives `requires_relationship_evidence` and `verified_relationship_count`. No "match", "verified match", or "meets criteria" language is permitted when relationship evidence is missing. The `sanitizeRelationshipMessage` function blocks dishonest match patterns. `verified_relationship_count` is currently always 0 (no relationship evidence extraction tool exists yet); the system is conservative by design. Tests in `relationship-predicate.test.ts` (16 tests).
- **ToolResult Contract**: Shared types and helpers for consistent data exchange between tools and the supervisor.
- **Google Places Query Mode**: `SEARCH_PLACES` supports two modes via `google_query_mode` field: `TEXT_ONLY` (query = `"${businessType} ${location} ${country}"`, no location bias params) and `BIASED_STABLE` (query = `"${businessType} ${country}"` + Google `location`/`radius`/`region` params for geo-biased search). Default is `TEXT_ONLY`. `BIASED_STABLE` resolves lat/lng from `geo-regions.ts` bounding box centroids first, falling back to Google Geocoding API, with in-memory cache. If geocoding fails, falls back to `TEXT_ONLY` with `bias_unavailable_fallback=true`. Every SEARCH_PLACES call returns a `SearchPlacesDebug` object (`google_query_mode_requested`, `google_query_mode_used`, `google_query_string`, `region_used`, `bias_applied`, `bias_location`, `radius_used`, `pages_fetched`, `results_returned`, `bias_unavailable_fallback`) surfaced in step artefact `outputs_summary.search_debug` for both v1 and replan paths. UK country variants auto-set `region=uk`.
- **Core Tools**:
    - `WEB_VISIT`: A deterministic website crawler and text cleaner.
    - `CONTACT_EXTRACT`: A deterministic contact detail extractor.
    - `WEB_SEARCH`: A strict, auditable web search fallback using Brave Search API.
    - `LEAD_ENRICH`: A deterministic lead pack builder.
    - `ASK_LEAD_QUESTION`: An evidence-backed question answerer.

### System Design Choices
- **IStorage Interface**: Provides an abstraction for database operations.
- **Logging Infrastructure**: Robust, structured logging integrated with Tower.
- **Concurrent Execution Handling**: In-memory progress tracking and a Map-based event system ensure isolated and efficient handling.
- **Error Handling**: Comprehensive error handling with proper status updates for failed plans.
- **Completion Gating**: Ensures `run_completed` is emitted only with Tower approval for `SEARCH_PLACES` runs.
- **Tower AFR Provability**: Every `SEARCH_PLACES` Tower call emits specific AFR events and a `tower_judgement` artefact.
- **Live Activity `clientRequestId` Threading**: `RunState` carries `clientRequestId` for correlation across all Tower events.
- **Mandatory Inline Tower Observation**: Always-on inline Tower observation with fatal failures.
- **Plan Executor Per-Step Tower Judgement**: The plan executor calls Tower via `judgeArtefact` after every step, enabling a bounded retry/replan inner loop.
- **Bypass Detector**: Detects runs that bypass the Supervisor.
- **Short-Circuit Diagnostics**: Emits `diagnostic` artefacts and AFR events when runs short-circuit due to concurrency guard, execution errors, or unhandled exceptions, ensuring at least one artefact is always visible for any agent run.
- **Canonical Run ID Propagation**: `startJob` for `deep_research` sets `run_id` as a column AND in `request_data`, using the caller-provided `run_id` or `jobId` as the canonical ID, preventing run ID mismatches between the routing layer and supervisor.
- **Batch-Claim + Background Claimer**: The supervisor atomically claims ALL pending tasks upfront before processing any, preventing the external Wyshbone UI from racing to grab tasks during long-running executions. A background claimer runs every 2 seconds (via `setInterval`) to claim tasks that arrive while the main loop is busy processing, queuing them for immediate processing after the current batch.
- **Attribute Verification Gate (Website-Only, Places-Only Mode)**: Verifies hard constraints (e.g., "live music") after `SEARCH_PLACES` using website-only strategy. Google Places Details is called per lead to fetch `website` and `phone` (logged as `places_details_websites_found=X/Y`). For attribute checks: visits `lead.website` directly via `WEB_VISIT` (max_pages=3, page_hints=['home','events','whats-on']). If no website from Places, verdict=unknown with `unknown_reason=no_website_from_places`. ZERO Brave/WEB_SEARCH usage in enrichment or attribute verification.
- **Learning Layer v1**: A deterministic policy engine using a canonical three-policy bundle (`radius_policy_v1`, `enrichment_policy_v1`, `stop_policy_v1`) with `policy_bundle_version: 1`. Stored in `policy_versions.policy_data` as the canonical JSON contract. A `GLOBAL_DEFAULT` scope key seeds defaults on first use. Every run writes a `policy_applications` row with the canonical snapshot (`scope_key`, `applied_at`, `applied_versions`, `applied_policies`, `why_short`). Legacy flat policies auto-upgrade via `upgradeFlatPolicyToBundle`. Execution params (`searchBudgetCount`, `searchCount`, `maxReplans`, `enrichmentBatchSize`, radius steps/caps) are derived from the bundle and override supervisor defaults. The `why_short` array includes a line for `stop_policy_v1.max_replans` stating whether it is default or learned and showing the previous value. The `policy_application_snapshot` artefact payload includes `applied_max_replans`. The Activity UI renders policy snapshot cards with `max_replans` badge and `why_short` lines.
- **Tool Planning Policy**: Deterministic tool ordering rules for primary and fallback paths.
- **Verification Pending Artefact**: When hard attribute verification starts (e.g., "beer garden", "live music"), a `verification_pending` artefact is emitted immediately after enrichment completes, showing total checks expected and attributes being verified. This keeps the UI informed during the 1-2 minute verification window.
- **Run Bridge Diagnostics**: `bridgeRunToUI` logs full request/response details (payload, status, body) and emits a `diagnostic` artefact on HTTP or network failure, including the response body and request payload for debugging.
- **Attribute Evidence Artefacts**: For every HAS_ATTRIBUTE hard constraint, the attribute verification loop emits one `attribute_evidence` artefact per lead+attribute pair. Each artefact contains `verdict` (yes/no/unknown), `confidence` (high/medium/low), `evidence` (source_url, quote, source_type), `match_source` (title/body/search_snippet), `rationale`, and the `attribute_key`. Uses a keyword mapping system (e.g., "live music" → ["live music", "live band", "open mic", ...]) and a trust ladder (official site with clear mention → high confidence yes; directory/guide → medium; weak hints → unknown low; official contradiction → no high).
- **Attribute Verification Resilience**: Keyword scan on page.title + page.text_clean, domain-root fallback for deep-path URLs, awaited artefact writes. Debug with `ATTR_VERIFY_TRACE=1` env flag.
- **Simultaneous Task+Message Writes**: The final message insert and supervisor_tasks status update are fired in parallel via `Promise.all` so the chat bubble and Activity panel resolve at the same time.

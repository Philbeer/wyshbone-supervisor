# Wyshbone Supervisor Suite

## Overview
Wyshbone Supervisor is a B2B lead generation system designed for automatic prospect identification and scoring. It delivers real-time lead suggestions with contact information via email and an integrated chat. The system aims to enhance sales processes, expand market reach, provide actionable data, and improve workflow efficiency. Future ambitions include integrating AI for plan evaluation, deep research, and intelligent replanning to optimize lead generation and verification.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### UI/UX Decisions
The frontend uses React, TypeScript, Vite, and Wouter. Styling is managed with Tailwind CSS and custom design tokens. UI components leverage `shadcn/ui` (based on Radix UI primitives) adopting a "New York" aesthetic inspired by Linear's B2B design.

### Technical Implementations
- **Frontend**: React with TypeScript, Vite, Wouter, and TanStack Query.
- **Backend**: Node.js with Express and TypeScript (ESM modules).
- **Data Storage**: PostgreSQL (Neon serverless) with Drizzle ORM.
- **Lead Generation Logic**: Employs conditional step execution, automatic data source fallback, and leverages historical performance data within a real-time, concurrent execution pipeline.
- **Chat Integration**: Uses a queue-based architecture with shared Supabase tables for AI interaction and intent-based routing.
- **Job Execution**: Manages background tasks such as nightly maintenance, syncing, monitoring, and lead generation with lifecycle management.
- **Agentic Decision Loop**: Integrates with the Tower Judgement API for plan evaluation (`CONTINUE`, `RETRY`, `CHANGE_PLAN`, `STOP`).
- **Logging**: A three-tier logging system provides comprehensive monitoring with structured logs.
- **Progress Tracking**: Uses an in-memory store for real-time tracking of concurrent plan executions.
- **Event System**: A Map-based registry per `planId` ensures isolated event streams during concurrent executions.
- **Deep Research**: A multi-provider research system generates reports based on user queries.
- **Intent Classification**: Routes messages with lead-finding verbs to `SEARCH_PLACES` and explicit deep research keywords to `DEEP_RESEARCH`.
- **Plan Execution**: Calls Tower via `judgeArtefact` after every step for step-level judgements. The final run verdict comes from a SINGLE authoritative Tower call on a `final_delivery` artefact, issued AFTER CVL verification completes. No Tower calls are made on `leads_list` or `attribute_verification` artefacts. The replan loop uses local count-based heuristics (`buildShortfallDirective`) instead of per-replan Tower calls. Invariant: `towerJudgedBeforeVerification = false` — Tower never judges leads before verification. If Tower returns STOP/FAIL on final_delivery, the verdict cannot flip to PASS.
- **Candidate Pool Expansion**: A deterministic planner heuristic in `server/supervisor/mission-planner.ts` (`computeCandidatePoolStrategy()`) that prevents premature verification on too-small candidate sets. When the plan includes discovery plus downstream verification/filtering constraints (`text_compare`, `website_evidence`, `relationship_check`, `attribute_check`, `status_check`, `ranking`), the candidate pool is expanded to `requested_count × 3`, capped at 30. If no count is specified, the pool defaults to 20 with no expansion (`applied=false`). If the user explicitly requests N results (e.g. 5), pool expands to min(N×3, 30) = 15. The `CandidatePoolStrategy` object (`{ requested_results, candidate_pool_size, multiplier, reason, applied }`) is stored in `MissionPlan.candidate_pool`, logged by `logMissionPlan()`, and used by `mission-executor.ts` to inflate `currentSearchBudget` and `effectiveEnrichBatch` (replacing the fixed `ENRICH_BATCH_SIZE=10`). A `candidate_pool_strategy` diagnostic artefact is emitted for AFR visibility. When no verification constraints exist (pure discovery), `applied=false` and pool equals the requested count — no inflation. Does not change execution loop, Tower, or UI.
- **Relationship Direction Reasoning**: `server/supervisor/relationship-direction.ts` implements a deterministic (no LLM) heuristic that decides which side of a relationship query is more likely to publish evidence. When a relationship constraint is detected (e.g. "organisations that work with the local authority"), the module identifies the left entity (what we're searching for) and right entity (the relationship target), scores each using `AUTHORITY_PATTERNS` (institutional entities like councils/NHS/universities score 3, large corporates/brands score 2, general businesses score 1), considers predicate directionality (`supplies` biases toward the right entity, `funded_by` biases toward the left), and chooses a direction: `reverse` (search from the authority/institutional entity first) or `forward` (standard). When reverse is chosen, `buildReverseSearchQueries()` generates targeted queries like `"${authority}" partners ${entityCategory}`. The direction is stored in `MissionPlan.relationship_direction` (optional `RelationshipDirectionResult`), logged in `logMissionPlan()`, and emitted as a `planner_relationship_direction` diagnostic artefact in `mission-executor.ts`. When direction is `reverse`, the executor's WEB_SEARCH phase uses the pre-built reverse queries instead of per-lead queries. Fallback: when both entities have similar scores (within threshold 1), standard forward search is used. This change only affects planning logic — execution loop, Tower judgement, and UI behaviour are unchanged.
- **Stage 2 Mission Planner**: `server/supervisor/mission-planner.ts` implements deterministic constraint-type → plan-strategy mapping for canonical `StructuredMission` input. Runs after Stage 1 extraction and emits a `mission_plan` artefact. Four core rules: **RULE_DISCOVERY** (entity + location → SEARCH_PLACES), **RULE_DIRECT_FIELD_CHECK** (text_compare/numeric_range → SEARCH_PLACES → FILTER_FIELDS, no web visits), **RULE_WEBSITE_EVIDENCE** (website_evidence/attribute_check/status_check → SEARCH_PLACES → WEB_VISIT → EVIDENCE_EXTRACT → TOWER_JUDGE), **RULE_RELATIONSHIP_EXTERNAL** (relationship_check → SEARCH_PLACES → WEB_SEARCH → WEB_VISIT → EVIDENCE_EXTRACT → TOWER_JUDGE, **never** discovery-only). Multiple constraints are sequenced cheapest-first (direct < website < external). Every plan includes: canonical constraints received, rules fired, tool sequence, verification method, selection reason. Tests in `mission-planner.test.ts` (20 tests).
- **Implicit Constraint Expansion**: `server/supervisor/implicit-constraint-expander.ts` adds a deterministic (no LLM) layer between Pass 1 and Pass 2 that detects implicit user intent from raw phrasing and the Pass 1 checklist. Supports four v1 cases: (1) ranking language ("best", "top", "highest rated") → infers soft ranking signals on `rating` and `review_count` with `prefer_high` operator; (2) website evidence language ("on their website", "site mentions") → reinforces `website_evidence`; (3) relationship language ("works with", "partnered with", "supplies", "serves", "affiliated with") → reinforces `relationship_check`; (4) name/text matching language ("in the name", "called", "named") → reinforces `text_compare`. All inferred constraints default to `soft` hardness unless the user clearly stated them as requirements. The result separates `explicit_constraints` (from checklist) and `inferred_constraints` (from phrasing patterns), with `inference_notes` for full observability. When inferences produce a semantic addendum, it is appended to the Pass 1 interpretation before feeding into Pass 2. `ImplicitExpansionTrace` type is in `mission-schema.ts`. `MissionExtractionTrace` carries `implicit_expansion`. A `implicit_constraint_expansion` diagnostic artefact is emitted in `mission-executor.ts` for AFR visibility. `MissionDiagnosticPayload.layers` includes the expansion trace. Shadow mode logging surfaces expansion details.
- **Pass 1 Constraint Checklist**: `server/supervisor/mission-extractor.ts` Pass 1 now outputs a JSON object with `constraint_checklist` (13 boolean fields classifying constraint types: `has_entity`, `has_location`, `has_text_compare`, `has_attribute_check`, `has_relationship_check`, `has_numeric_range`, `has_time_constraint`, `has_status_check`, `has_website_evidence`, `has_contact_extraction`, `has_ranking`, `has_requested_count`, `has_monitoring_intent`) and `semantic_interpretation` (the clean English text passed to Pass 2). The checklist is purely observational — it does NOT change execution, schema, or planning. A `pass1_constraint_checklist` diagnostic artefact is emitted in `mission-executor.ts` for AFR visibility. The `ConstraintChecklist` type is defined in `mission-schema.ts`. `MissionExtractionTrace` carries `pass1_constraint_checklist`. `MissionDiagnosticPayload.layers` includes the checklist. Shadow mode logging in `mission-bridge.ts` surfaces active checklist flags. Graceful fallback: if Pass 1 returns plain text instead of JSON, the raw text is used as the semantic interpretation and checklist is null.
- **Mission-Driven Execution (Live)**: `server/supervisor/mission-executor.ts` implements `executeMissionDrivenPlan()` which is now the PRIMARY execution path when mission extraction succeeds. The `MissionPlan` built by `buildMissionPlan()` directly drives execution — search params are derived from `StructuredMission` (entity_category, location_text, constraints), NOT from the legacy `ParsedGoal`. The executor follows the plan's `tool_sequence` through phases: SEARCH_PLACES → FILTER_FIELDS → Evidence Gathering (WEB_VISIT/WEB_SEARCH per lead) → Tower semantic verification → RANK_SCORE → Replan on shortfall (radius expansion via RADIUS_LADDER_KM) → Final Tower judgement on `final_delivery` artefact → Delivery summary emission. Returns the same shape as `executeTowerLoopChat`: `{ response, leadIds, deliverySummary, towerVerdict, leads }`. The `ParsedGoal` downgrade via `missionToParsedGoal()` still happens for compatibility (monitor creation, clarify/constraint gates), but it is no longer the source of truth for execution. `executeTowerLoopChat` is retained as a legacy fallback that fires ONLY when mission extraction or plan building fails. The branch is controlled by `useMissionExecution` flag set in supervisor.ts after successful `buildMissionPlan()`. Observability: `execution_source: 'mission' | 'legacy'` is logged at the branch point and included in all artefact payloads. Full artefact trail: `mission_plan` → `step_result` (per phase) → `attribute_verification` (evidence) → `leads_list` → `final_delivery` → `tower_judgement` → `delivery_summary`.
- **Two-Phase Multi-Tool Execution**: Discovers leads with `SEARCH_PLACES` then enriches them using `WEB_VISIT`, `CONTACT_EXTRACT`, and `LEAD_ENRICH`, with `WEB_SEARCH` as a fallback.
- **Attribute Evidence Investigation Loop**: When HAS_ATTRIBUTE constraints require verification, the system uses a three-tier investigation strategy instead of only checking cached enrichment data. (1) `cached_pages`: uses WEB_VISIT pages from the enrichment phase; (2) `active_web_visit`: if a lead has a website but no cached pages (wasn't in the enrichment batch), actively calls WEB_VISIT to fetch and scan the site; (3) `web_search_then_visit`: if a lead has no website from Places, calls WEB_SEARCH to find a URL, then WEB_VISIT to scan it. Budget limits: `ATTR_MAX_ACTIVE_VISITS` (default 8) caps active WEB_VISIT calls, `ATTR_MAX_SEARCH_FALLBACKS` (default 5) caps WEB_SEARCH fallback calls. Each `attribute_evidence` artefact carries `investigation_strategy` and the summary `attribute_verification` artefact includes `investigation_breakdown` counts. A reusable `scanPagesForAttribute()` helper handles keyword matching identically across all three paths. `activeWebVisit()` and `webSearchForAttribute()` helper functions handle the tool calls with proper artefact creation and error handling.
- **RESTful API**: Manages leads, user context, signals, and plan execution, including creation, approval, and progress monitoring.
- **Constraint Verification Layer (CVL)**: An additive layer that extracts, checks, and performs per-lead deterministic verification, influencing the final verdict. `CATEGORY_EQUALS` constraints are **disabled** — business type is used as a search query term only, never as a verifiable constraint. The `PLACES_SUPPORTED_CATEGORIES` whitelist in `cvl.ts` is retained as dead code but never triggered because `c_category` constraints are filtered out at parse time and no longer safety-injected by the supervisor. This prevents niche business types (e.g. "micropubs") from being classified as `blocking_hard` and triggering CVL override stops. For `HAS_ATTRIBUTE` constraints, the CVL receives an `AttributeEvidenceMap` (keyed by placeId → attribute → {verdict, confidence, reason, evidenceUrl}) from the attribute verification gate, so per-lead attribute verdicts (yes/no/unknown) are incorporated into `verified_exact` counts and the CVL override logic. `LOCATION_EQUALS` uses geo bounding box verification via `server/supervisor/geo-regions.ts` — leads carry `lat`/`lng` from Google Places, and verification checks whether the point falls inside known region bounding boxes. Composite regions (e.g. "Sussex" = East Sussex + West Sussex) are supported. CVL uses a truth-preserving four-status model: `VerificationStatus = 'yes' | 'no' | 'unknown' | 'search_bounded'`. For SOFT location constraints, `search_bounded` counts toward `verified_exact` (the search was bounded to the region); for HARD constraints, only `yes` (VERIFIED_GEO) satisfies. Each lead carries a `location_confidence: LocationConfidence` field (`verified_geo`, `search_bounded`, `out_of_area`, `unknown`, `not_applicable`). Summary includes `location_breakdown: { verified_geo_count, search_bounded_count, out_of_area_count, unknown_count }` and per-constraint `leads_search_bounded` counts.
- **Intelligent Replanning**: Separates user-requested counts from search budgets, accumulates and deduplicates leads across replan versions, employs a progressive geographic expansion strategy, enforces hard constraints, and includes early stopping. When no user count is specified, the system performs a single Google page 1 search (budget=20), skips count-based replanning, and treats all returned results as complete (no shortfall, no "missing" messaging). Count-based replan overrides, Learning Layer budget caps, and `requested_count` hard constraints only activate when the user explicitly specifies a number. The supervisor separates `searchBudgetCountFromGoal` (internal budget, 30–50 for search) from `userRequestedCountFinal` (user-facing cap). All user-facing artefacts (`leads_list`, `final_delivery`, `dsLeads` for delivery summary) are capped to `userRequestedCountFinal` before emission. A `count_diagnostic` artefact is emitted after every `leads_list` artefact, logging `userRequestedCountFinal`, `searchBudgetCount`, `candidateCountFromGoogle`, and `deliveredCountAfterTrim` for pipeline transparency.
- **Canonical Requested Count**: All runs carry a `RequestedCountCanonical` object (`requested_count_user: "explicit" | "any"`, `requested_count_value: number | null`, `requested_count_effective: number`). `DEFAULT_LEADS_TARGET = 20`. When user specifies no count → `{user: "any", value: null, effective: 20}`. Every Tower `successCriteria` includes all three fields plus `target_count = requested_count_effective` (never null). This eliminates Tower STOPs caused by missing count fields. Defined in `goal-to-constraints.ts` via `buildRequestedCount()`. Regression tests in `requested-count.test.ts`.
- **LLM-backed Goal-to-Constraints Parser**: Converts natural language user goals into structured constraints using LLMs and strict JSON schema validation. Country is inferred from location (US states → "US", UK locations → "UK") rather than defaulting to "UK". The `inferCountryFromLocation` function provides regex fallback country detection. `sanitiseLocationString()` strips instruction contamination from location fields using `LOCATION_INSTRUCTION_PATTERNS` (a shared pattern array also used by `stripInstructionClauses`) covering "and include email/phone/website", "including email", "with email", parenthesised forms, and comma/semicolon variants. A keyword-based fallback guard (`INSTRUCTION_KEYWORD_RE`) catches residual contamination by splitting on commas to extract the city-only prefix. In the supervisor, `searchQuery.location` is only used as a fallback when `parsedGoal.location` is empty, and is always sanitised first — preventing LLM-polluted overrides (e.g. "arundel and include email"). `detectExactnessMode()` detects hard vs soft exactness intent. `detectDoNotStop()` detects "do not stop" phrases (logged, ignored — budgets enforced instead). `stripInstructionClauses()` removes instruction text from raw goals before constraint extraction. `detectDeliveryRequirements()` detects "include email/phone/website" as delivery requirements, never contaminating location. `ParsedGoal` includes `include_email`, `include_phone`, `include_website` boolean fields. Contact requests (email/phone/website) are tracked as `contactRequests` in the supervisor and passed to artefacts as `contact_requests` metadata with explicit notes that they are enrichment-only, NOT search or pass/fail constraints. The plan artefact's `original_user_goal` uses `normalizedGoal` (sanitised) instead of `originalUserGoal` (raw) to prevent Tower from interpreting delivery requirements as hard constraints.
- **Tower Semantic Attribute Verification**: `server/supervisor/tower-semantic-verify.ts` provides per-lead, per-attribute semantic verification via Tower. After WEB_VISIT extracts evidence (snippets from `text_clean`), the Supervisor calls Tower at `POST /api/tower/semantic-verify` with `{ run_id, original_user_goal, lead_name, lead_place_id, constraint_to_check, source_url, evidence_text, extracted_quotes, page_title }`. Tower returns `{ status: 'verified' | 'weak_match' | 'no_evidence' | 'insufficient_evidence', confidence: number, reasoning: string, matched_snippets?: string[] }`. The Supervisor does NOT make semantic verdicts — Tower does. WEB_VISIT success alone does NOT mark a lead as verified. **Step-status suppression**: When a run has hard HAS_ATTRIBUTE constraints (`willRunSemanticVerification` flag), the enrichment-phase `judgeArtefact()` call and `tower_judgement` artefact for WEB_VISIT are suppressed — these only prove tool execution, not attribute truth, and create misleading "Tower Judgement: pass (WEB_VISIT)" entries in the AFR timeline. Step-status Tower checks remain active for CONTACT_EXTRACT, LEAD_ENRICH, and all non-attribute queries. Supervisor action rules: `verified` → keep as verified (verdict=yes, high confidence); `weak_match` → keep but label weak (verdict=yes, low confidence); `no_evidence` → not verified (verdict=unknown); `insufficient_evidence` → unverified, optionally try another source. A separate `tower_semantic_judgement` artefact is created per lead per attribute. The `attribute_evidence` artefact includes `verification_source: 'tower_semantic' | 'keyword_only' | 'no_evidence'` and `tower_semantic: { status, confidence, reasoning }`. The `attribute_verification` summary includes `tower_semantic_summary: { tower_called, tower_verified, tower_weak_match, tower_no_evidence, keyword_only_fallback }`. Stub mode (TOWER_ARTEFACT_JUDGE_STUB=true) returns plausible stub results based on evidence availability. Falls back gracefully if Tower is unavailable.
- **Constraint-Led Quote Extraction (v3 — Hardened Evidence Layer)**: `server/supervisor/constraint-led-extractor.ts` implements a deterministic (no LLM) extraction layer between WEB_VISIT and Tower. **v3 evidence hardening** (5 improvements): **(1) Tiered phrase matching**: `scoreSentence()` classifies matches into tiers: `exact_value` (12pts), `full_phrase`/`synonym` multi-word (8pts), single-word synonym (5pts), `entity_expansion` multi-word (7pts) / single (4pts), `relationship_predicate` (3pts), `single_word` fragment (1pt). Single-word fragments from multi-word constraint values score only 1pt and require local context support (nearby related words in ±4 word window) to upgrade. Basic stemming via `basicStem()` handles -ing, -ed, -tion, -ment, -ness, -ies, -s suffixes. Stop words filtered from individual word targets. Minimum score threshold raised to 2 (3 for relationships). **(2) Relationship evidence tightening**: `scoreRelationshipSentence()` is a dedicated scorer for `relationship_check` constraints. Requires BOTH entity mention AND relationship predicate in same sentence for top score (14pts if <100 chars apart, 10pts otherwise). Entity-only mention scores only 3pts (7pts if a named entity pattern like "Blackpool Council" detected). Predicate-only scores 1pt. Generic mentions like "we support councils" no longer produce high-confidence evidence. **(3) Constraint-specific fallback**: `mission-executor.ts` now runs per-constraint fallback searches (up to 3) instead of one shared search. Each uses `getPageHintsForConstraint()` to add targeted terms. `CONSTRAINT_VALUE_PAGE_HINTS` maps constraint values to URL patterns (live music→events/whats-on, vegan→menu/dining, spa→spa/wellness, relationship→partners/case-studies). **(4) Source-aware confidence**: `SOURCE_CONFIDENCE_MULTIPLIERS` — website=1.0, gov_page=1.15, search_snippet=0.35, social_media=0.5, directory=0.45. `PAGE_CLASS_MULTIPLIERS` — evidence_rich=1.2, neutral=1.0, noise=0.5. `classifyPageType()` detects evidence-rich pages by URL patterns matched to constraint value. Privacy/cookie/terms pages classified as noise. **(5) Enhanced page bias**: `EVIDENCE_PAGE_BIAS` expanded with spa/wellness/gym/fitness/rooftop/brew/garden patterns. `CONSTRAINT_VALUE_PAGE_HINTS` provides per-value URL biasing. `scorePageRelevance()` awards 10pts for evidence-rich pages, -5 for noise, +8 for title containing constraint value. WEB_VISIT now uses `page_hints` from `getPageHintsForConstraint()` (up to 6 hints) and crawls up to 5 pages. Context snippet radius expanded to ±120 chars. Sorting tiebreaker uses match tier rank. SYNONYM_MAP expanded to 24 entries (added rooftop, spa, craft beer, brew, personal training, swimming pool, gym, conference, wedding, sustainability). All backward-compatible fields preserved.
- **Real Execution Timeout System**: Both `mission-executor.ts` and legacy `supervisor.ts` enforce execution timeouts via `checkDeadline()` / `checkRunDeadline()`. Default timeout is `RUN_EXECUTION_TIMEOUT_MS_DEFAULT = 120_000` (120s). Configurable via `RUN_EXECUTION_TIMEOUT_MS` env var (falls back to legacy `MAX_RUN_DURATION_MS`). Tool call limit: `MAX_TOOL_CALLS_PER_RUN` (default 150). When a deadline is exceeded: (1) `runDeadlineExceeded` flag is set with a reason string; (2) verdict is forced to `'timeout'`, action to `'stop'`; (3) `agent_runs.status` is set to `'timed_out'` (not `'completed'`), `terminalState` = `'timed_out'`; (4) metadata includes `timed_out: true` and `timeout_reason`; (5) a `diagnostic` artefact titled "Run timed out" is emitted with elapsed_ms, tool_calls, limits, and leads_at_timeout. Stale sweep (`sweepOrphanedAgentRuns()`) and startup recovery (`recoverOrphanedAgentRuns()`) both mark stuck `executing` runs as `timed_out` (previously `failed`). The `evaluateAndRecoverTask()` function recognises `timed_out` as a terminal status alongside `completed` and `failed`.
- **Evidence Extraction Step**: After WEB_VISIT for attribute queries, the `extractEvidenceSnippets()` function in `server/supervisor.ts` performs sentence-level evidence extraction from `text_clean`. Text is split into sentences (by `.!?` boundaries), each scored by keyword relevance (match count, position, sentence length), and the top 1-3 snippets are stored. Fallback to 200-char chunk extraction if no sentences detected. The `attribute_evidence` artefact now includes: `extracted_quotes: string[]` (sentence-level snippets from page body, not just page titles), `page_url`, `page_title`, `extraction_method` (`keyword_sentence_match` | `keyword_window` | `no_match` | `none`), `original_goal`, `constraint_raw`. The `evidence.quote` field is updated to prefer the best extracted sentence over the old narrow keyword-window snippet. `run-receipt.ts` prefers `extracted_quotes[0]` over `evidence.quote` for snippet display.
- **Run Receipt Artefact**: `server/supervisor/run-receipt.ts` emits a `run_receipt` artefact after every completed run. This is the canonical, user-facing "what happened" summary that the UI can render directly. The receipt matches `lead_pack` and `contact_extract` artefacts to **delivered leads only** (by place_id, then normalised name, then title substring) and computes proven contact counts. CONTACT_EXTRACT artefacts are discovered from two sources: legacy `type: 'contact_extract'` artefacts AND `type: 'step_result'` artefacts where `payload.step_type === 'CONTACT_EXTRACT'` (the production format). Contacts are extracted from `payload.outputs.contacts`, `payload.contacts`, or `payload.contact_extract_outputs.contacts`. Step-result CONTACT_EXTRACT artefacts carry `lead_place_id` and `lead_name` in their payload for robust matching. `contacts_proven` is `false` if no contact artefacts could be matched — in that case, counts are `null` and the narrative uses safe language ("Contact details varied by venue"). **Attribute outcomes**: `run_receipt.outcomes.attributes` (optional, present when `attribute_evidence` or `attribute_verification` artefacts exist) summarises HAS_ATTRIBUTE verification results per attribute. Each entry has `attribute_raw`, `matched_count`, `matched_place_ids`, `unknown_count`, and `evidence_refs` (with `place_id`, `url`, optional `snippet` and `matched_variant`). Only `verdict=yes` with a non-empty `evidence.source_url` counts as matched; everything else is unknown. Primary source: `attribute_evidence` artefacts; fallback: `attribute_verification` artefact's `results[]` array. Narrative lines include attribute verification summaries. The `narrative_lines` array contains non-technical chat-bubble text. A `debug` object includes `artefact_ids_used` (with `lead_pack`, `contact_extract`, and `attribute_evidence` arrays), `counting_method`, and `notes`. Emitted after `emitDeliverySummary` and before goal status update. Tests in `run-receipt.test.ts` (15 tests).
- **Search Query Compiled Artefact**: `server/supervisor/search-query-compiled.ts` emits a `search_query_compiled` artefact at end of every run, capturing `interpreted_location`, `interpreted_query`, `requested_count`, `exactness_mode`, `do_not_stop_ignored`, radius escalation stats, candidate counts from Google, `stop_reason`, `query_broadening_applied`, and `query_broadening_terms` for full pipeline transparency.
- **Synonym Broadening (Underfill Recovery)**: `QUERY_SYNONYM_MAP` and `getQueryBroadeningTerms()` in `replan-policy.ts` provide synonym expansion (e.g. "pubs" → "pubs OR bar OR inn") as a last-resort underfill recovery when radius expansion is exhausted and no progress is being made. Broadening is tracked via `queryBroadeningApplied`/`queryBroadeningTerms` in the supervisor and surfaced in the `search_query_compiled` artefact.
- **Canonical Delivery Summary**: `delivery_summary` provides a canonical `status` field (`COMPLETED`|`STOP`|`ERROR`) and `trust_status` field (`TRUSTED`|`UNTRUSTED`) derived from delivered lead count, serving as the authoritative user-facing output. `deriveCanonicalStatus()` uses a simple rule: if leads were delivered (`totalDelivered > 0`), status is `COMPLETED` regardless of Tower verdict; if no leads and Tower error, status is `ERROR`; otherwise `STOP`. The Tower verdict (`pass`/`fail`/`stop`) is preserved in `tower_verdict` field as metadata but does NOT determine the delivery summary status. Terminal state in `agent_runs` is always `completed` (never `stopped`) when the pipeline reaches final delivery. `final_delivery` payload includes `requested_count`, `delivered_count`, and `found_count_raw`; leads are trimmed to `requested_count` before final delivery to prevent over-delivery. `TRUSTED` = COMPLETED; `UNTRUSTED` = STOP or ERROR. The `cvl_summary` field includes `location_breakdown: { verified_geo_count, search_bounded_count, out_of_area_count, unknown_count }` for honest UI bucketing. The `leads` artefact payload attaches per-lead `verification: { location_status, location_confidence, verified_exact, all_hard_satisfied, verification_level }` so the UI can render verification status without inference. `verification_level` is `'verified'` (CVL checks + exact), `'checked'` (CVL checks + not exact), `'candidate'` (CVL match but 0 checks), or `'unverified'` (no CVL match). `verified_exact` is never `true` unless CVL actually ran checks (vacuous-truth guard in `verifyLeads`). Classification into `delivered_exact[]` vs `delivered_closest[]` uses CVL per-lead verification for location constraints when available, falling back to substring heuristics only when CVL data is absent. `delivered_exact_count` always equals `delivered_exact.length` — no count/array mismatch. Final message metadata includes `final_verdict` (canonical status) and `trust_status` for downstream consumers.
- **Canonical Intent Wiring (Semantic Overreach Reduction)**: When the intent extractor produces a valid `CanonicalIntent` (shadow mode active), downstream gates trust its structured fields instead of re-parsing raw user messages with regex. `evaluateClarifyGateFromIntent()` uses `mission_type` for routing; `preExecutionConstraintGateFromIntent()` builds constraint contracts from `intent.constraints[]`; `classifyFollowUp()` accepts optional `followUpIntent` for scope-change detection; `evaluatePreflightClarify()` uses canonical `entity_category`/`location_text`/time constraints. All paths fall back to legacy regex when canonical intent is unavailable or `mission_type=unknown`. ### System Design Choices
- **IStorage Interface**: Provides an abstraction for database operations.
- **Logging Infrastructure**: Robust, structured logging integrated with Tower.
- **Concurrent Execution Handling**: In-memory progress tracking and a Map-based event system ensure isolated and efficient handling.
- **Error Handling**: Comprehensive error handling with proper status updates for failed plans.
- **Completion Gating**: Ensures `run_completed` is emitted only with Tower approval for `SEARCH_PLACES` runs.
- **Tower AFR Provability**: Every `SEARCH_PLACES` Tower call emits specific AFR events and a `tower_judgement` artefact.
- **Live Activity `clientRequestId` Threading**: `RunState` carries `clientRequestId` for correlation across all Tower events.
- **Mandatory Inline Tower Observation**: Always-on inline Tower observation with fatal failures.
- **Plan Executor Per-Step Tower Judgement**: The plan executor calls Tower via `judgeArtefact` after every step, enabling a bounded retry/replan inner loop.
- **Bypass Detector**: Detects runs that bypass the Supervisor.
- **Short-Circuit Diagnostics**: Emits `diagnostic` artefacts and AFR events when runs short-circuit due to concurrency guard, execution errors, or unhandled exceptions, ensuring at least one artefact is always visible for any agent run.
- **Canonical Run ID Propagation**: `startJob` for `deep_research` sets `run_id` as a column AND in `request_data`, using the caller-provided `run_id` or `jobId` as the canonical ID, preventing run ID mismatches between the routing layer and supervisor.
- **Batch-Claim + Background Claimer**: The supervisor atomically claims ALL pending tasks upfront before processing any, preventing the external Wyshbone UI from racing to grab tasks during long-running executions. A background claimer runs every 2 seconds (via `setInterval`) to claim tasks that arrive while the main loop is busy processing, queuing them for immediate processing after the current batch.
- **Orphaned Agent Runs Recovery**: On startup, `recoverOrphanedAgentRuns()` finds `agent_runs` stuck in `executing` state for > 3 minutes and marks them as `failed` with `orphan_reason: 'server_restart'` in metadata. This handles runs that were interrupted by server restarts where the `supervisor_tasks` recovery couldn't help (task already completed/failed but agent_run left in executing). The `sweepStaleTasks` and `recoverOrphanedTasks` both now check for `claimed` status in addition to `processing`.
- **Canonical Intent Extractor (Active Primary Parser)**: `server/supervisor/canonical-intent.ts` defines the `CanonicalIntent` Zod schema — a single source of truth for extracted user intent with enum-enforced fields: `mission_type`, `entity_kind`, `entity_category`, `location_text`, `geo_mode`, `radius_km`, `requested_count`, `default_count_policy`, `constraints[]` (each with `type`, `raw`, `hardness`, `evidence_mode`, `clarify_if_needed`, `clarify_question`), `plan_template_hint`, `preferred_evidence_order`. `server/supervisor/intent-extractor.ts` provides `extractCanonicalIntent(msg, conversationContext?)` — calls OpenAI gpt-4o-mini or Anthropic claude-3-5-haiku with an upgraded strict JSON-only prompt containing 4 few-shot examples, detailed field rules, negative rules, and conversation context handling. `server/supervisor/intent-shadow.ts` implements the feature flag `INTENT_EXTRACTOR_MODE` (off|shadow|active|strict). **The canonical extractor is now the ACTIVE primary parser**: when extraction is valid, `canonicalIntentToParsedGoal()` builds a full `ParsedGoal` from `CanonicalIntent` (mapping entity_category→business_type, location_text→location, canonical constraints→StructuredConstraint[], geo_mode→LOCATION_NEAR/LOCATION_EQUALS, inferring country from location). - **Early Intent & Constraints Artefact Emission**: `processChatTask()` now runs `parseGoalToConstraints()` and emits `constraints_extracted` + `constraint_capability_check` artefacts BEFORE the clarify gate and constraint gate. This ensures that even when the clarify gate routes to `clarify_before_run` or `direct_response` (returning early), AFR still shows what constraints were extracted from the user's message. The parsed goal is passed through to `executeTowerLoopChat()` via the `preComputedParsedGoal` parameter, which reuses it (avoiding a duplicate LLM call) and skips re-emitting the artefacts. - **Schema**: `StructuredMissionSchema` includes `entity_category`, `location_text`, `requested_count` (number|null — extracted only when user explicitly states a count), `constraints[]`, `mission_mode`.
- **Two-Pass Architecture**: Pass 1 extracts semantic meaning in plain English (no JSON). Pass 2 maps the semantic interpretation into the fixed `StructuredMission` JSON schema. Both passes use GPT-4o-mini or Claude 3.5 Haiku.
- **Intent Source Priority** (active mode): `mission` (active+valid) → `canonical` → `legacy`. When `getMissionExtractorMode() === 'active'` and mission extraction succeeds, `missionToParsedGoal()` is the PRIMARY source. If mission bridge fails, falls through to canonical intent. If canonical also fails, falls through to legacy regex parser. `intentSource` is `'mission' | 'canonical' | 'legacy'`. `extraction_method` in artefacts: `structured_mission_extractor` | `canonical_intent_extractor` | `llm_parsed_goal` | `regex_fallback_deprecated`.
- **Feature Flag**: `MISSION_EXTRACTOR_MODE=active` (default), `shadow`, or `off`. In `active` mode, valid missions drive planning. In `shadow` mode, missions are extracted and logged but canonical/legacy drive planning.
- **Observability**: Full `MissionExtractionTrace` captures raw input, pass 1 interpretation, pass 2 structured mission, validation result, model, and timing for both passes. `logMissionShadow()` compares mission output with legacy `CanonicalIntent` and logs differences (runs in both active and shadow modes). `MissionFailureStage` type (`none`, `no_api_key`, `pass1_llm_call`, `pass2_llm_call`, `pass2_json_parse`, `pass2_schema_validation`, `extractor_exception`) pinpoints exactly where failures occur. Every run emits a `mission_extraction` artefact (persisted to DB) and a `diagnostic` POST to UI with `diagnostic_type: 'mission_extraction'`. Artefact summary includes `mode=active|shadow`. `legacyFallbackReason` is tracked when higher-priority sources fail and propagated to `constraints_extracted` and intent preview payloads.
- **Completeness Check**: `server/supervisor/mission-completeness-check.ts` — deterministic post-extraction audit that detects when concept categories (`relationship`, `time`, `website_evidence`, `status`, `ranking`) appear in raw input or pass1 semantic interpretation but are absent from the structured mission. Hard drops (relationship, time, website_evidence) → `block` (mission invalidated, forces fallback); soft drops (status, ranking) → `clarify`. Integrated after successful mission extraction in `supervisor.ts`; results logged to AFR and persisted as `mission_completeness_check` artefacts. Non-fatal (wrapped in try/catch).
- **Bridge**: `missionToParsedGoal()` converts `StructuredMission` to existing `ParsedGoal` format for downstream compatibility. All 11 constraint types mapped with **rich type preservation**: `text_compare` → `NAME_CONTAINS`/`NAME_STARTS_WITH`; `attribute_check` → `HAS_ATTRIBUTE`; `relationship_check` → `RELATIONSHIP_CHECK`; `status_check` → `STATUS_CHECK`; `time_constraint` → `TIME_CONSTRAINT`; `website_evidence` → `WEBSITE_EVIDENCE`; `ranking` → `RANKING`; `numeric_range` → `COUNT_MIN`; `contact_extraction` → sets `include_email`/`include_phone`/`include_website` flags (no constraint emitted); `entity_discovery` → skipped (covered by `entity_category`); `location_constraint` → `LOCATION_NEAR` (with default 10km radius) or `LOCATION_EQUALS`. Each mapped constraint carries an optional `canonical` field preserving the original `MissionConstraint` data. `requested_count` from mission maps to `requested_count_user`, `search_budget_count` (min(max(30, count*3), 50)), and `success_criteria.target_count`.
- **Rich Constraint Types**: `goal-to-constraints.ts` extended with `RELATIONSHIP_CHECK`, `STATUS_CHECK`, `TIME_CONSTRAINT`, `WEBSITE_EVIDENCE`, `RANKING` in `CONSTRAINT_TYPES`. `ATTRIBUTE_LIKE_TYPES` array lists all types that behave like attribute checks for downstream verification. `isAttributeLikeConstraint(type)` replaces raw `=== 'HAS_ATTRIBUTE'` checks throughout the codebase (in `supervisor.ts`, `query-shape-key.ts`, `mission-bridge.ts`, `intent-bridge.ts`). `StructuredConstraint` schema includes optional `canonical` field for lossless round-tripping.
- **Handoff Diagnostic**: `buildHandoffDiagnostic()` in `mission-bridge.ts` compares canonical mission constraints against mapped `StructuredConstraint[]` and detects semantic downgrades (e.g., relationship→HAS_ATTRIBUTE). Returns `HandoffDiagnostic` with `mapping_fidelity` (`exact` | `degraded`), `downgrades[]` with reason/detail, and canonical/mapped constraint summaries. Emitted as `mission_handoff_diagnostic` artefact and UI diagnostic after every successful mission bridge.
- **Clarify Gate**: Monitoring/alert verbs (`monitor`, `check`, `checking`, `watch`, `alert`, `notify`, `track`) added to `LEAD_FINDING_VERBS` so monitoring requests always route to `agent_run`. Dedicated `hasMonitoringIntent()` helper with `MONITORING_VERBS` regex detects `keep checking`, `monitor`, `watch for`, `alert me`, `notify me`, `track`, `let me know`, `keep an eye`, `ongoing`, `recurring`, `check every`. Monitoring detection is the FIRST check in both `evaluateClarifyGateFromIntent()` (checks `mission_type=monitor` then `hasMonitoringIntent` regex) and `evaluateClarifyGate()` (checks `hasMonitoringIntent` regex) — early exit to `agent_run` before any other classification logic can interfere. This prevents downstream `explain`/`meta_question`/`isDirectResponse` checks from misrouting monitoring requests.
- **Monitoring Routing**: `'monitor'` added to `MISSION_TYPE_ENUM` in `canonical-intent.ts`. Intent extractor prompt includes `mission_type` classification rules and two monitoring examples. Supervisor creates `scheduled_monitors` row when `mission_mode` is `monitor`, `alert_on_change`, or `recurring_check` after successful tower loop execution. Monitor creation emits `monitor_created` artefact. Response is appended with monitoring confirmation note. All monitor creation is non-fatal (failures are logged but don't block the response).

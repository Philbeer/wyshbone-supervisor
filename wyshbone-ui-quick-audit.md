## 2. Database Tables (from `shared/schema.ts`)

### Agent/Supervisor Related
| Table | Purpose |
|-------|---------|
| `plans` | Lead generation plans with status (pending_approval, executing, completed, failed) |
| `plan_executions` | Historical record of plan runs with step results |
| `supervisor_state` | Checkpoint tracking for signal processing (timestamp, last ID) |
| `processed_signals` | Idempotency tracking - which signals have been processed |
| `subconscious_nudges` | Background "nudges" generated by subcon packs (stale leads, etc.) |

### CRM/Lead Related
| Table | Purpose |
|-------|---------|
| `users` | Basic user auth (id, username, password) |
| `user_signals` | User activity signals (search_performed, profile_update, idle) |
| `suggested_leads` | AI-generated lead suggestions with rationale, score, lead data |

## 4. Integration Points

### External Services (via Environment Variables)

| Service | Env Vars | Purpose |
|---------|----------|---------|
| **Supabase** | `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE` | User data, signals, messages, conversations, supervisor_tasks |
| **Google Places** | `GOOGLE_PLACES_API_KEY` | Business discovery |
| **Hunter.io** | `HUNTER_API_KEY` or `HUNTER_IO_API_KEY` | Email finding |
| **Resend** | `RESEND_API_KEY`, `RESEND_FROM_EMAIL` | Email notifications |
| **PostgreSQL** | `DATABASE_URL` | Local plan storage |
| **Control Tower** | `TOWER_URL`, `TOWER_API_KEY` | Execution logging |

### Supabase Tables Read/Written

**Read from Supabase:**
- `users` - Profile, account_id, vertical_id
- `user_signals` - Trigger lead generation
- `facts` - User knowledge facts
- `messages` - Conversation history
- `conversations` - Conversation metadata
- `supervisor_tasks` - Pending tasks from UI
- `scheduled_monitors` - Active monitors
- `deep_research_runs` - Research history

**Written to Supabase:**
- `messages` - Supervisor responses
- `supervisor_tasks` - Status updates

### Cross-Repo Integration

The Supervisor is designed to integrate with **Wyshbone-UI** via:
1. **Supabase `supervisor_tasks` table** - UI creates tasks, Supervisor processes
2. **Supabase `messages` table** - Supervisor writes responses with `source='supervisor'`
3. **Realtime subscriptions** - UI listens for new Supervisor messages

See `INTEGRATION_INSTRUCTIONS_FOR_WYSHBONE_UI.md` for full integration guide.

## 5. Agent Features

### Action Types (Canonical Capabilities)
```typescript
type ActionType = 
  | 'DEEP_RESEARCH'       // Deep research on topics
  | 'GLOBAL_DB'           // Business/lead database search
  | 'SCHEDULED_MONITOR'   // Set up recurring monitoring
  | 'EMAIL_FINDER';       // Find emails for leads
```

### Lead Generation Plan Flow
1. **Plan Creation** (`planLeadGeneration()`)
   - Parses goal into structured format
   - Generates step DAG (Google Places → Hunter Domain → Hunter Enrich → Save List)
   - Optionally adds email sequence and monitoring steps

2. **Plan Execution** (`executeLeadGenerationPlan()`)
   - Executes steps in dependency order
   - Retry logic with exponential backoff
   - Conditional branching (SUP-010)
   - Fallback data sources (SUP-011)
   - Historical performance bias (SUP-012)

3. **Progress Tracking**
   - Real-time status updates
   - Step-by-step results
   - Persisted to `plan_executions` table

## 6. Key Concepts

### SupervisorService (`server/supervisor.ts`)
- Polls Supabase every 30 seconds
- Processes `user_signals` → generates leads
- Processes `supervisor_tasks` → responds to chat requests
- Monitors goals for issues

### Subconscious Packs (`server/subcon/`)
- Background analyzers that run on schedule
- Currently: `stale_leads` pack
- Produce "nudges" for users

### Verticals (`server/verticals/`)
- Industry-specific configurations
- Currently: `brewery` vertical
- Determines which features/packs are enabled

### Event Bus (`server/core/event-bus/`)
- In-memory pub/sub for internal events
- Used for feature execution lifecycle

## 8. Architecture Notes

- **Monorepo style** but single-deployable unit
- **Express backend** with TypeScript
- **React frontend** with Vite + Tailwind
- **Drizzle ORM** for PostgreSQL
- **Supabase** for shared state with UI repo
- **Polling architecture** (not webhooks) for Supabase integration
- **Feature flags** via environment variables

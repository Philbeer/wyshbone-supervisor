Goal
Add a safe "implicit constraint expansion" layer between Pass 1 and Pass 2 so Wyshbone can convert common loose user phrasing into explicit structured constraints without relying on brittle manual phrasing rules.

This is the next improvement after the Pass 1 constraint checklist.

Current pipeline

User input
→ Pass 1 constraint checklist + semantic interpretation
→ Pass 2 structured mission
→ schema validation
→ mission planner
→ mission executor

We are adding a small new layer:

User input
→ Pass 1 constraint checklist + semantic interpretation
→ implicit constraint expansion
→ Pass 2 structured mission
→ schema validation
→ mission planner
→ mission executor

IMPORTANT
This must be conservative and production-grade.
Do NOT invent strong constraints recklessly.
Do NOT redesign planner, executor, or Tower.
Do NOT change StructuredMission schema unless absolutely necessary.
This is primarily an interpretation-quality upgrade.

------------------------------------------------
WHAT THIS LAYER SHOULD DO
------------------------------------------------

Some user queries imply extra useful structure even when they do not state it in exact schema language.

Examples:

1. "best dentists in Brighton"
This implies ranking intent.
It may also imply helpful ranking signals such as:
- reviews / rating
- prominence / quality signals
But these should be added only as soft inferred signals, not hard facts.

2. "pubs with Swan in the name"
This implies text_compare on name.

3. "pubs that mention live music on their website"
This implies website_evidence.

4. "organisations that work with the local authority"
This implies relationship_check.

The new layer should:
- detect these implicit meanings from the Pass 1 output
- expand them into explicit candidate constraints/signals
- mark which ones are inferred vs explicit
- keep inferred constraints soft unless the user clearly made them hard

------------------------------------------------
IMPLEMENTATION
------------------------------------------------

Add a small module, for example:

server/supervisor/implicit-constraint-expander.ts

It should take as input:
- raw user input
- Pass 1 constraint checklist
- Pass 1 semantic interpretation

It should return something like:

{
  "explicit_constraints": [...],
  "inferred_constraints": [...],
  "inference_notes": [...]
}

Important distinction:
- explicit_constraints = clearly stated by user
- inferred_constraints = helpful structure inferred from user wording
- inferred constraints must not silently become hard blocking requirements unless explicitly justified

------------------------------------------------
FIRST VERSION: SUPPORT THESE CASES ONLY
------------------------------------------------

1. Ranking language
Examples:
- best
- top
- highest rated

Behaviour:
- create or reinforce a ranking signal
- optionally infer soft ranking signals such as review_count / review_score / prominence
- keep these soft

2. Website evidence language
Examples:
- on their website
- mentions on its site
- says on website

Behaviour:
- reinforce website_evidence explicitly

3. Relationship language
Examples:
- works with
- partnered with
- supplies
- serves
- affiliated with

Behaviour:
- reinforce relationship_check explicitly

4. Name/text matching language
Examples:
- in the name
- called
- named
- with X in the title

Behaviour:
- reinforce text_compare on name

------------------------------------------------
SAFETY RULES
------------------------------------------------

1. Never turn vague user wording into strong hard constraints unless the user clearly intended that.
2. Inferred constraints/signals should default to:
- soft
- advisory
- ranking support
not hard blocking filters.

3. Preserve observability.
We must be able to see:
- what was explicit
- what was inferred
- why it was inferred

4. If no safe inference can be made, do nothing.

------------------------------------------------
ARTEFACT / OBSERVABILITY
------------------------------------------------

Emit a diagnostic artefact called:

implicit_constraint_expansion

Payload example:

{
  "explicit_constraints": [...],
  "inferred_constraints": [
    {
      "type": "ranking",
      "field": "review_score",
      "operator": "prefer_high",
      "value": null,
      "hardness": "soft",
      "source": "inferred_from_best_language"
    }
  ],
  "inference_notes": [
    "Detected ranking language: 'best'"
  ]
}

This artefact should appear in AFR.

SUCCESS CRITERIA
------------------------------------------------

Query:
"Find the best dentists in Brighton"

Expected:
- explicit or reinforced ranking constraint
- optional soft inferred ranking signals
- no invented hard factual requirement

Query:
"Find pubs in Arundel with Swan in the name"

Expected:
- text_compare reinforced on name
- no random extra inferences

Query:
"Find pubs in Arundel that mention live music on their website"

Expected:
- website_evidence reinforced

Query:
"Find organisations that work with the local authority in Blackpool"

Expected:
- relationship_check reinforced
- no unrelated inferred constraints

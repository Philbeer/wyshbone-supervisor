CONTEXT
After the re-loop chain completes, the combined-delivery Tower call (around
line 763) judges the merged delivery from all loops. This call has two
problems:

1. It receives deliveredLeads filtered by entity.verified === true. The
   entity.verified flag is set at wrong granularity in both adapters
   (gp-cascade-adapter.ts:41 ties it to the whole loop's Tower verdict;
   gpt4o-adapter.ts:49 hardcodes it to true). Neither reflects per-lead
   verification.

2. It re-judges leads that per-loop Tower already judged. When per-loop
   Tower passed for at least one loop, the combined call risks producing
   a contradictory verdict (e.g. per-loop Loop 2 PASS, combined FAIL).

The current skip condition is too narrow — it only skips for single-loop
runs that passed:

  const skipCombinedTower = totalLoops === 1 && perLoopTowerVerdict === 'pass';

This leaves the combined call active for ANY multi-loop run, including the
cascade-then-web-search pattern where Loop 1 fails (cascade) and Loop 2
passes (web search). The combined call then fails wrongly, the response
builder honestly reads the bad verdict, and the user sees "0 verified"
when 15 leads were delivered.

GOAL
Trust per-loop Tower verdicts as source of truth. Skip the combined Tower
call whenever at least one per-loop Tower already passed. Use the best
per-loop verdict as the final combined verdict.

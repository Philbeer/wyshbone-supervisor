Audit the repository to verify that the Stage 1 chat interpretation upgrade was implemented correctly.

Context

We just implemented a new chat interpretation system with:

- a fixed constraint taxonomy
- a two-pass LLM extraction pipeline
- deterministic schema validation
- removal/reduction of regex intent parsing
- observability outputs for interpretation

Your job is NOT to change anything yet.

Your job is to perform a forensic audit and explain exactly what was implemented.

Return a clear report.

------------------------------------------------

PART 1 — Architecture

Explain the current pipeline for interpreting user chat.

Trace the exact execution path:

user message
→ pass 1 semantic interpretation
→ pass 2 structured mission extraction
→ validation
→ mission passed to planner

Show which files and functions implement each stage.

------------------------------------------------

PART 2 — Pass 1 Meaning Extraction

Identify:

• the prompt used
• where the LLM is called
• the function that executes it
• the structure of the pass-1 output

Confirm whether the output is a **semantic interpretation** rather than copying user text.

Example input to test mentally:

find pubs in arundel that have the word swan in the name

Explain what Pass 1 should output.

------------------------------------------------

PART 3 — Pass 2 Schema Mapping

Show where pass-2 mapping happens.

Confirm that the result produces a structured mission object with fields like:

entity_category
location_text
constraints[]
mission_mode

Explain exactly how constraint types are mapped.

------------------------------------------------

PART 4 — Constraint Taxonomy

List all supported constraint types currently implemented.

Expected set:

- entity_discovery
- location_constraint
- text_compare
- attribute_check
- relationship_check
- numeric_range
- time_constraint
- status_check
- website_evidence
- contact_extraction
- ranking

Confirm whether the code enforces this taxonomy.

------------------------------------------------

PART 5 — Regex Audit

Find all remaining regex that attempt to interpret natural language intent.

Report:

• file
• line
• purpose

Classify each one as:

SAFE
cleanup only

OR

RISKY
still interpreting language

------------------------------------------------

PART 6 — Validation Layer

Show where the structured mission is validated.

Explain:

• schema validation
• failure behaviour
• fallback behaviour

Confirm that invalid constraint types or malformed fields cannot enter the planner.

------------------------------------------------

PART 7 — Observability

Confirm that the system logs or emits:

• raw_user_input
• semantic_interpretation_pass1
• structured_mission_pass2

Show where these are emitted or stored.

------------------------------------------------

PART 8 — End-to-End Test

Simulate what the system would produce for these inputs:

1.
find pubs in arundel that have the word swan in the name

2.
find pubs in arundel that mention live music on their website

3.
find breweries in texas opened in the last 6 months

4.
keep checking which hospitals in the UK offer the sleep apnea implant and alert me when it becomes available near me

For each case show the expected structured mission.

------------------------------------------------

PART 9 — Remaining Gaps

Identify anything that still violates the Stage 1 design.

Examples:

• regex still doing interpretation
• LLM copying surface phrasing
• constraint taxonomy not enforced
• observability missing

------------------------------------------------

PART 10 — Final Verdict

Give a score out of 10 for the Stage 1 implementation.

Explain:

• what is correct
• what still needs improvement
• whether the system is now safe to use for planning.

Return the report only.

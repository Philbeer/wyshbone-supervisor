We are now doing Evidence Layer Hardening before building the replanning loop.

Goal

Strengthen the current evidence layer so that Tower receives stronger, more trustworthy, more constraint-specific evidence.
We do NOT want to build a great recovery loop on top of weak evidence extraction.
This task is to improve the evidence system itself first.

Context

A code audit of the current evidence layer showed:

- the architecture already exists
- constraint-led extraction already exists
- quote-based artefacts already exist
- relationship direction reasoning already exists
- fallback logic already exists

So this is NOT a rebuild.

But the audit also showed these weaknesses:

1. Phrase matching is too weak
   - single words like "music", "beer", "bars" match too easily
   - full phrases are not weighted strongly enough
   - false positives are too common
   - synonym coverage is limited
   - no stemming / weak normalisation

2. Relationship proof is too loose
   - seeing "council" on a page can count as weak evidence
   - relationship checks do not strongly require:
     - target entity
     - relationship predicate
     - tight local context
   - this creates weak or misleading relationship evidence

3. Fallback sourcing is too shallow
   - only one fallback web search
   - shared across all constraints
   - often too broad
   - weak websites quickly become weak evidence

4. Source weighting is too crude
   - official source vs snippet vs directory is not strongly separated enough

5. Candidate/page bias is underused
   - the extractor is not yet strongly biasing toward likely evidence-rich pages like:
     - /events
     - /menu
     - /spa
     - /partners
     - /case-studies
     - /clients

We want to harden the evidence layer before moving to replanning.

Implementation tasks

TASK 1 – Tighten phrase matching quality

Improve phrase matching so that:
- exact full-phrase matches are strongly preferred
- multi-word phrase matches score much higher than loose word overlap
- weak single-word matches are reduced heavily or ignored unless supported by local context
- basic normalisation/stemming is added where sensible
- exact match / synonym match / weak lexical overlap are clearly distinguished

Important:
Do not make this benchmark-specific.
This must remain generic and operate from constraint objects.

Desired effect:
Fewer false positives and stronger quotes for Tower.

---

TASK 2 – Tighten relationship evidence requirements

For relationship_check constraints, evidence must be stronger.

Do not treat vague mentions of:
- council
- NHS
- trust
- partner
as meaningful proof by themselves.

Require stronger relationship evidence, ideally:
- target entity mention
- relationship predicate
- same sentence or tight context window
- preferably both parties named if available

Examples:

Good:
"We work with Blackpool Council to deliver community services."

Weak:
"We support councils and communities."

Good:
"We supply software to NHS trusts across the region."

Weak:
"We understand NHS data standards."

The extractor should score these very differently.

Desired effect:
Local authority / NHS / supplier relationship checks stop producing misleading weak positives.

---

TASK 3 – Improve fallback sourcing

Current fallback is too shallow.

Make fallback constraint-specific rather than one broad fallback shared across all constraints.

Fallback order should become:

1. alternate likely pages on same domain
2. targeted web search for the specific constraint
3. broader fallback evidence source if needed

Examples:
- live music → /events /whats-on /entertainment pages
- vegan → /menu /food /dining pages
- spa → /spa /wellness /facilities pages
- relationships → /partners /clients /case-studies /frameworks pages

Fallback evidence must be clearly labelled as weaker when appropriate.

Desired effect:
Blocked or weak websites do not immediately collapse evidence quality.

---

TASK 4 – Strengthen source-aware evidence weighting

Improve evidence weighting by source type.

Higher confidence sources:
- official website pages
- official authority / council / NHS / gov pages
- explicit partner/client/case-study pages

Lower confidence sources:
- search snippets
- directories
- generic marketing pages
- social media unless explicitly useful

Desired effect:
Tower receives evidence with more realistic confidence structure.

---

TASK 5 – Improve candidate/page evidence bias

Use the existing page bias concept more effectively.

Before scanning everything equally, bias toward pages more likely to contain constraint-relevant evidence.

Examples:
- /events for live music
- /menu for vegan
- /spa /facilities for spa
- /personal-training /services for PT
- /partners /clients /case-studies /frameworks for relationship checks

This is not replanning.
This is just improving first-pass evidence gathering.

Desired effect:
Better first-pass evidence quality without changing overall architecture.

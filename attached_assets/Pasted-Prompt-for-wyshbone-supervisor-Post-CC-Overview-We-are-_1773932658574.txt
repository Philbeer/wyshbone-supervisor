Prompt for: wyshbone-supervisor-Post-CC
Overview
We are implementing the Universal Re-Loop Architecture. This adds a loop skeleton that wraps the existing GP cascade and GPT-4o search execution paths. Instead of a linear "run one executor → judge → deliver" flow, the system now runs: Trigger → Planner → Executor → Judge → Gate → (re-loop or deliver).
The GP cascade (executeMissionDrivenPlan with executionPath='gp_cascade') and GPT-4o search (executeGpt4oPrimaryPath) stay completely untouched internally. We wrap each with a thin adapter conforming to a shared executor interface. The loop skeleton calls these adapters.
CRITICAL RULES

Do NOT modify server/supervisor/gpt4o-search.ts internally — only import from it
Do NOT modify the GP cascade logic inside server/supervisor/mission-executor.ts — we are adding new files and a new entry point in mission-executor that delegates to the loop skeleton
The existing executeMissionDrivenPlan and executeGpt4oPrimaryPath functions must continue to work exactly as they do today — they become the internal implementations behind the executor adapters
All new code goes in new files. The only existing file that gets modified is mission-executor.ts (to add the new loop-aware entry point) and storage.ts / schema.ts (to add the loop_state table)

2. server/supervisor/reloop/executor-registry.ts
A simple registry that maps executor type IDs to their adapter functions. This is where adding a new tool (e.g. DataLedger API) is a single registration call.
typescriptimport type { ExecutorInput, ExecutorOutput } from './types';

export type ExecutorFn = (input: ExecutorInput) => Promise<ExecutorOutput>;

const registry = new Map<string, ExecutorFn>();

export function registerExecutor(executorType: string, fn: ExecutorFn): void {
  registry.set(executorType, fn);
  console.log(`[EXECUTOR_REGISTRY] Registered executor: ${executorType}`);
}

export function getExecutor(executorType: string): ExecutorFn | undefined {
  return registry.get(executorType);
}

export function getAvailableExecutors(): string[] {
  return Array.from(registry.keys());
}
3. server/supervisor/reloop/gp-cascade-adapter.ts
A thin wrapper around the existing executeMissionDrivenPlan (GP cascade path). It calls the existing function with executionPath: 'gp_cascade' and maps the result to ExecutorOutput.

Import executeMissionDrivenPlan and the types it needs from mission-executor.ts
Reconstruct the MissionExecutionContext from the ExecutorInput (the missionContext field carries the full original objects — StructuredMission, MissionPlan, IntentNarrative etc)
Call executeMissionDrivenPlan(ctx) with executionPath: 'gp_cascade'
Map the MissionExecutionResult to ExecutorOutput:

entities = the leads array mapped to ExecutorEntity
entitiesAttempted = total candidate count (from rawResult payload if available)
executionMetadata = extracted from the artefacts/logs
coverageSignals.maxResultsHit = true if Google Places returned its cap
rawResult = the full MissionExecutionResult for Tower/delivery


Register this adapter in the executor registry as 'gp_cascade'

Important: The existing function already handles the full pipeline (search → filter → evidence → replan → Tower judge → delivery summary). For the re-loop architecture, we need the adapter to return the results before the final delivery packaging, so the loop skeleton can accumulate across loops. However, to avoid modifying the existing function, the adapter should:

Call the existing function as-is (it does its own Tower judge call internally)
Extract the entities and metadata from the result
Return them as ExecutorOutput
The loop skeleton's judge step will evaluate the accumulated results across all loops, not just this loop's output — it will make its own Tower call on the combined set

This means there will be two Tower judge calls for the first loop (one inside the executor, one from the loop skeleton). That's acceptable for now — the executor's internal Tower call is part of its existing behaviour that we're not changing. The loop skeleton's judge call evaluates the accumulated picture.

5. server/supervisor/reloop/planner.ts
Rules-based planner (v1). Receives the trigger context and returns which executor to use.
Rules:

First loop, no preference: Use gp_cascade (cheap, fast, structured data)
Re-loop after GP exhaustion: Use gpt4o_search (broader web coverage)
Re-loop after GPT exhaustion: No more executors available → signal deliver (let gate force delivery)
User explicitly chose GPT-only (executionPath === 'gpt4o_primary'): Use gpt4o_search on first loop
Circuit breaker active: Return whatever's fastest with a note that this is the last attempt

The planner function signature:
typescriptexport function plan(context: {
  loopNumber: number;
  loopHistory: LoopRecord[];
  executionPath?: 'gp_cascade' | 'gpt4o_primary';
  availableExecutors: string[];
  circuitBreaker: boolean;
}): PlannerDecision
Log the reasoning: [RELOOP_PLANNER] Loop {n}: chose {executor} because {reason}
6. server/supervisor/reloop/gate.ts
Rules-based gate (v1). Reads the judge verdict and decides re-loop vs deliver.
Rules:

PASS verdict + confidence > 0.6: stop_deliver
CAPABILITY_FAIL + another executor available that hasn't been tried: re_loop with suggestedNextExecutor
PARTIAL + another executor available: re_loop if coverage < 60%, else stop_deliver
EXECUTION_FAIL: re_loop once (retry same executor), then stop_deliver
Circuit breaker (loop >= MAX_LOOPS): stop_deliver with circuitBreaker: true
No improvement between loops (entities_found same or fewer): stop_deliver — don't waste API calls

MAX_LOOPS = 3 (configurable via env RELOOP_MAX_LOOPS)
The gate builds contextForward with accumulated entities (deduped by name across all loops), the full loop history, and a human-readable failureContext string that explains why we're going again. This string is critical — it's what the planner (and eventually the LLM planner) reads to decide the next strategy.
Log: [RELOOP_GATE] Loop {n}: decision={decision} reason={reason} accumulated={count} circuitBreaker={bool}
7. server/supervisor/reloop/judge-adapter.ts
Translates the Tower's existing verdict into the re-loop JudgeVerdict format. This does NOT replace the Tower — it wraps the Tower's response and adds the variable state analysis.

After the executor runs, the adapter examines the executor's output to build VariableState:

resultCount: compare entities found vs requested count
toolExhaustion: check coverageSignals.maxResultsHit
coverageGap: if requested 20 and got 5, coverage = 25%
evidenceQuality: count verified vs total
duplicateRate: compare against knownEntities from previous loops


The verdict is derived from the variable state:

All green (enough results, decent evidence): PASS
Tool maxed out, significant coverage gap: CAPABILITY_FAIL
Errors or rate limits: EXECUTION_FAIL
Some results but below threshold: PARTIAL


The recommendation combines the verdict with what executors remain available
The rawTowerVerdict preserves the Tower's original verdict from inside the executor

8. server/supervisor/reloop/loop-skeleton.ts
The main loop orchestrator. This is the core of the re-loop architecture.
typescriptexport async function runReloop(params: {
  runId: string;
  userId: string;
  conversationId?: string;
  clientRequestId?: string;
  rawUserInput: string;
  mission: StructuredMission;
  plan: MissionPlan;
  missionTrace: MissionExtractionTrace;
  intentNarrative: IntentNarrative | null;
  queryId?: string | null;
  executionPath?: 'gp_cascade' | 'gpt4o_primary';
}): Promise<MissionExecutionResult>
Flow:

Generate a chain_id (uuid) for this loop chain
Build the initial ExecutorInput from the params
Loop start:
a. Call planner.plan() to get the executor choice
b. Look up the executor in the registry
c. Pass knownEntities from previous loops (accumulated entity names)
d. Call the executor adapter
e. Call judgeAdapter.evaluate() on the output
f. Call gate.decide() on the verdict
g. Persist to Supabase: Write a loop_state row with the full loop record (planner decision, executor output summary, judge verdict, gate decision, entity counts)
h. Create an artefact: type: 'reloop_iteration' with title Re-loop iteration {n}: {executorType} → {verdict} containing the loop record summary
i. Log AFR event: reloop_iteration with loop number, executor, verdict, gate decision
j. If gate says re_loop: update knownEntities, increment loop number, continue loop
k. If gate says stop_deliver: break
After loop exits:

The final executor's rawResult contains the full MissionExecutionResult (including delivery summary, Tower verdict, leads)
Return that directly — the existing delivery pipeline in the executor already packaged everything
Create a final artefact: type: 'reloop_chain_summary' with title Re-loop complete: {totalLoops} loops, {totalEntities} entities summarising the full chain



Deduplication across loops: Maintain a Map<string, ExecutorEntity> keyed by normalised name (lowercase, trimmed, strip "The " prefix). When loop 2 returns entities, only add those not already in the map. Pass the accumulated map's keys as knownEntities to subsequent executors.
Circuit breaker: After MAX_LOOPS iterations, force stop_deliver regardless of judge verdict. Set circuitBreaker: true. Create a diagnostic artefact noting the circuit breaker fired.

typescriptexport async function executeMissionWithReloop(
  ctx: MissionExecutionContext,
): Promise<MissionExecutionResult> {
  // Feature flag: RELOOP_ENABLED (default 'true')
  const reloopEnabled = (process.env.RELOOP_ENABLED || 'true').toLowerCase() === 'true';

    intentNarrative: ctx.intentNarrative,
    queryId: ctx.queryId,
    executionPath: ctx.executionPath,
  });
}

## Logging standards

All re-loop logs use the prefix `[RELOOP_*]`:
- `[RELOOP_SKELETON]` — loop lifecycle events
- `[RELOOP_PLANNER]` — planner decisions
- `[RELOOP_GATE]` — gate decisions
- `[RELOOP_JUDGE]` — judge evaluations
- `[RELOOP_EXECUTOR]` — executor adapter events
- `[RELOOP_PERSIST]` — Supabase writes

Goal: Fix Chat Gate G6. If user asks for vague subjective criteria and or an unrecognised location (eg “find the best vibes near council things”), the system must NOT present as runnable. It must ask explicit clarification questions and must not allow “Search now” to run until answered.

What’s currently wrong (evidence):

Router correctly goes to clarify_for_run and flags unrecognised location, vague/subjective entity type.

But the clarify response is generic (“a couple of quick questions”) and the clarify_state ends up showing “Ready to search” even though it isn’t.

The clarify session summary also stores nonsense like “the best vibes in council things” as if it’s valid.

Required behaviour (PASS criteria for G6):
When input contains:

a subjective / unmeasurable criterion (eg “best vibes”, “nice”, “good”, “best”, “cool”, “fun”) AND or

a location that is not a valid place (eg “council things”)
Then the assistant must ask 2 explicit questions:

Location: “What location should I search (city, town, postcode)?”

Criteria definition: “What does ‘best vibes’ mean to you?” with a short list of measurable proxies (nightlife, quiet, live music, cosy cafes, family friendly, etc.)
And the session must remain in “gathering / blocked” state until both are answered.

Implementation instructions (minimal, production grade):

In the clarify for run pipeline (where you currently decide missing fields), introduce a concept of blocking_missing_fields that can include:

location_invalid_or_missing

criteria_unmeasurable

Add detection for subjective criteria in the initial user message:

If message contains patterns like: best vibes|vibes|nice|good|cool|fun|best|great atmosphere|good atmosphere

Then set criteria_unmeasurable=true unless user already provided a measurable proxy (examples: “live music”, “quiet”, “late night”, “cocktails”, “beer garden”, “dog friendly”, “good for studying”).

Add detection for invalid location:

If location extractor returns a value but it does NOT look like a place (fails your existing looksLikeLocation style check), treat as missing/invalid.

Do NOT store it as the session location.

Build clarify questions explicitly from the blockers:

If location_invalid_or_missing: ask “Which location should I search? (city/town/postcode)”

If criteria_unmeasurable: ask “What does ‘best vibes’ mean to you? Choose one or two: lively nightlife, cosy/quiet, live music, good for studying, family friendly, scenic views, etc.”

Ensure the clarify session only transitions to “ready_to_search” when:

entity type present AND

valid location present AND

criteria_unmeasurable is false (ie user selected measurable proxies OR removed the subjective requirement)

Make sure “search now / execute now” is blocked if either blocker is present:

If user types “search now” while criteria_unmeasurable or location invalid: respond with a clear message listing what’s still needed.

Add or update tests:

New test: input “find the best vibes near council things”

must return clarify ask_more

must include missing fields for location + criteria

must not mark ready_to_search

New test: after providing location and measurable vibe proxy, it becomes runnable.

Deliverables:

Code changes only in Supervisor clarify logic (wherever clarify_for_run session + state is built).

Tests passing.

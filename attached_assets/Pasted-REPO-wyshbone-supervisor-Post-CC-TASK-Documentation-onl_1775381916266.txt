CURRENT ARCHITECTURE (use this as the source of truth):

**Intent Extraction Pipeline (3-tier fallback):**
1. **Mission Extractor (PRIMARY)** — `server/supervisor/mission-extractor.ts`. Three-pass LLM pipeline: Pass 1 (semantic interpretation) → Pass 3 (intent narrative with findability, scarcity, exclusions) → Pass 2 (structured JSON schema mapping). Always runs. Produces `StructuredMission` with typed constraints, `IntentNarrative` for search strategy, and full extraction trace. This is the active primary pipeline that drives all execution.

2. **Canonical Intent Extractor (OPT-IN FALLBACK)** — `server/supervisor/intent-extractor.ts` + `server/supervisor/intent-shadow.ts`. Controlled by env var `INTENT_EXTRACTOR_MODE` (default: `off`). When enabled (`active`), runs a separate LLM call producing a `CanonicalIntent` schema. Used as fallback if mission extraction fails. When `off` (default), does not run and costs nothing. The "shadow" name is historical — it was originally added as a diagnostic observer before the mission extractor was built.

3. **Legacy Goal Parser (EMERGENCY FALLBACK)** — `server/supervisor/goal-to-constraints.ts`. Single LLM call producing a `ParsedGoal`. Only runs if both mission and canonical extractors fail. Last resort.

The intent source is tracked per-run as `intentSource: 'mission' | 'canonical' | 'legacy'` and logged in artefacts for diagnostics.

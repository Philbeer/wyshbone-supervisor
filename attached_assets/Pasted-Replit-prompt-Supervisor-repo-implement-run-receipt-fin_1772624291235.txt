Replit prompt (Supervisor repo): implement run_receipt final artefact

You are working in the Wyshbone Supervisor repo.

Goal: Create a single final artefact called run_receipt (or delivery_receipt) at the end of every completed run. This artefact is the canonical, user-facing “what happened” summary that the UI can render without doing fragile cross-artefact counting.

This is NOT a UI task. Build the backend artefact only.

Why

Right now the UI re-derives counts (emails, phones, websites checked) by scanning multiple artefact types (lead_pack, contact_extract, etc). This is fragile and caused false statements like “0 emails found” or wrong counts. We want a backend-produced receipt that is always truthful.

Non-negotiable honesty rules

The receipt must never claim “0 emails/phones found” unless that was proven by counting actual extraction artefacts.

If contact counts cannot be proven, receipt must mark them as unknown and provide a safe human line like: “Contact details varied by venue.”

Receipt creation timing

Create/write the receipt:

after final_delivery is produced (and delivered leads are known)

before (or alongside) any final status / completion marker

Artefact contract

Create a new artefact type:

type = "run_receipt"

phase = "final" (or similar consistent field you use)

title like: Run receipt: pubs in Arundel (10 requested)

payload must include structured fields below

Required fields in run_receipt.payload_json

Identity

run_id

goal (the user goal string if available)

mission_type (if you have it)

created_at

Counts

requested_count (number)

delivered_count (number)

candidate_count_from_google (number if known)

plans and replans (numbers if you track)

Delivered lead references

delivered_leads: array of { name, place_id } (or your canonical lead id fields)

Website + extraction coverage (web scraping example)

websites_checked_count (number, proven)

website_missing_count (number, proven)

contact_extraction_attempted_count (number, proven)

These must be derived from the actual run artefacts / step outputs, not guessed.

Proven contact counts (website extraction only, ignore Google Places phones for now)

contacts_proven: boolean

contact_sources_used: string[] (e.g. ["lead_pack","contact_extract"])

unique_email_count: number | null

unique_phone_count: number | null

email_list_sample: string[] OPTIONAL (redacted or partial ok)

phone_list_sample: string[] OPTIONAL

Rules:

Only count emails/phones that came from lead_pack and/or contact_extract artefacts for the delivered leads only.

If you cannot link contact artefacts to delivered leads reliably, set contacts_proven=false and both counts null.

Human narrative (brief, non-tech)
Add narrative_lines: string[] designed to be shown directly in the chat bubble, like:

“I checked the websites of the 10 delivered pubs where available for contact details.”

“I found 5 public emails and 9 phone numbers from those websites.” (only if proven)

Otherwise: “Contact details varied by venue.”

No internal jargon like “artefacts”, “Tower”, “constraint_results”, “search_bounded”.

Diagnostics for dev (behind UI ‘show details’)

Find where final_delivery is produced and saved.

Write the new artefact row using your existing artefact storage mechanism.

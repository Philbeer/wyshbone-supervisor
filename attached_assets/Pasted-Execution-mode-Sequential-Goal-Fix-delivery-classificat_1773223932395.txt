Execution mode: Sequential

Goal
Fix delivery classification so weak or broken evidence cannot be promoted into delivered_exact for hard website-evidence queries.

Context

A real AFR run for:
"Find 5 pubs in Arundel that mention live music on their website"

showed this behaviour:

- 3 pubs had genuine website evidence for live music
- 2 additional pubs were padded in to reach the requested count
- all 5 were placed into delivered_exact
- all 5 were marked match_level: "exact" and match_valid: true
- internally they were only weak_match, not true verified evidence

This is wrong.

For hard website-evidence constraints:
- delivered_exact must contain only genuinely verified results
- weak_match must not be treated as exact delivery
- if only 3 verified pubs exist, the system must be allowed to deliver 3 honestly

Do not work on UI in this task.

Required changes

1. Find the logic that decides:
   - delivered_exact
   - delivered_closest
   - match_level
   - match_valid

2. Change the rules so that for hard evidence-required constraints:
   - only leads with true verified evidence go into delivered_exact
   - weak_match / partial / ambiguous evidence must go into delivered_closest
   - broken evidence attachment must not qualify as exact

3. Make shortfall an honest valid outcome:
   - if 3 verified results exist and user asked for 5, the system should be able to deliver 3 verified exact results and stop honestly
   - do not pad delivered_exact with weaker candidates just to hit count

4. Add regression tests covering:
   - 3 verified + 2 weak candidates
   - expected result: delivered_exact = 3, delivered_closest = 2
   - no weak candidates promoted to exact
   - no forced padding of exact results to requested count

Include:
- root cause

- how shortfall is now handled
- test coverage added

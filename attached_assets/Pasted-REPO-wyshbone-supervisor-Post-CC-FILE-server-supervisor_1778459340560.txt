CONTEXT
The gp_cascade adapter currently hardcodes evidence: [] on every entity it returns. This means the combined-delivery Tower judge sees leads with verified=true but no supporting evidence, fires the VERIFIED_WITHOUT_EVIDENCE gap, and returns STOP. The Tower is correct — the evidence has been gathered upstream but is being stripped at this boundary.

The sibling file gpt4o-adapter.ts (same directory) already implements the correct pattern: it pulls match_evidence (or supporting_evidence) from result.deliverySummary.delivered_exact by matching lead name, and attaches that array to each entity. We need gp-cascade-adapter.ts to do the same.

CHANGE
Replace the entire entities mapping block in gp-cascade-adapter.ts. Find this block:

  const entities: ExecutorEntity[] = result.leads.map(lead => ({
    name: lead.name,
    address: lead.address,
    phone: lead.phone,
    website: lead.website,
    placeId: lead.placeId,
    source: 'gp_cascade',
    verified: result.towerVerdict === 'pass' || result.towerVerdict === 'accept',
    verificationStatus: result.towerVerdict ?? 'unknown',
    evidence: [],
  }));

Replace it with:

  const entities: ExecutorEntity[] = result.leads.map(lead => {
    const dsLeads = (result.deliverySummary as any)?.delivered_exact || [];
    const dsLead = dsLeads.find((dl: any) => dl.name === lead.name);
    const matchEvidence = dsLead?.match_evidence || dsLead?.supporting_evidence || [];

    return {
      name: lead.name,
      address: lead.address,
      phone: lead.phone,
      website: lead.website,
      placeId: lead.placeId,
      source: 'gp_cascade',
      verified: result.towerVerdict === 'pass' || result.towerVerdict === 'accept',
      verificationStatus: result.towerVerdict ?? 'unknown',
      evidence: matchEvidence,
    };
  });

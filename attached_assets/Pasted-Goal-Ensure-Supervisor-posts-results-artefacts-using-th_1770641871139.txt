Goal: Ensure Supervisor posts results artefacts using the UI canonical runId, so AFR "View results" works for supervisor delegated runs.

Context:
UI "View results" fetches artefacts by runId. For supervisor delegated runs, the UI runId and the supervisor internal runId can diverge. We must avoid that. The canonical runId must be the one generated by UI and passed in supervisor_tasks.request_data.run_id.

Required changes:
A) Treat request_data.run_id as canonical runId for all artefact POSTs to UI.
- Everywhere Supervisor posts to POST /api/afr/artefacts, set runId = request_data.run_id (UI runId).
- Keep supervisor internal id in metadata, like metadata.supervisor_run_id or metadata.deep_research_run_id.

B) Ensure request_data always contains:
- run_id (UI runId)
- client_request_id (CRID)
If missing, emit an AFR event artefact_post_failed with reason "missing_identifiers" and do not silently succeed.

C) After Supervisor starts a deep research run (or similar background run that has its own internal id):
- Call the UI bridge endpoint (POST /api/afr/run-bridge) to update agent_runs with supervisor_run_id = the supervisor internal run id.
- This makes CRID lookups and debugging easier, but the canonical runId for artefacts is still the UI runId.

D) Observability:
- Keep the [ARTEFACT_POST] log line.
- Add a log line showing BOTH ids:
  uiRunId=<request_data.run_id> supervisorRunId=<internal id> clientRequestId=<crid>
- Ensure artefact_post_succeeded only fires when UI returns artefactId.

Acceptance tests:
1) Insert a supervisor_task with request_data.run_id and request_data.client_request_id.
2) Even on error path (like missing Google Places key), Supervisor must POST an artefact under the UI runId.
3) In the UI domain, /api/afr/artefacts?runId=<ui runId> must return the posted artefact.

Deliverables:
- Implement changes with full file rewrites where needed (no partial snippets).
- Keep internal ids only in metadata.
- Do not change UI schema, just align ids and bridge.

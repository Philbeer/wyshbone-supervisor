# ENHANCEMENT: Verify entity type тАФ don't trust Google Places classification

## REPO: wyshbone-supervisor-Post-CC

## CONTEXT

Google Places returns wrong entity types, especially for long-tail searches, edge cases, or queries with few results. When searching for "pubs", it returns breweries, wine bars, and restaurant-bars. When searching for "hairdressers", it returns barbers, beauty salons, and training academies. The fewer results available, the more aggressively it stretches.

The current system trusts Google Places classification and only verifies constraints (cask ale, independent, etc.). It does NOT independently verify that the result is actually the entity type the user asked for. A brewery passes the "serves cask ale" check because of course it does тАФ but it's not a pub.

The fix is to make entity type verification part of the evidence pipeline. The system already has everything it needs тАФ it visits the business's website, reads the content, and judges evidence. It just needs to also ask: "Is this actually a [pub/hairdresser/restaurant]?"

## WHAT TO CHANGE

### Change 1: Pass 3 key_discriminator тАФ include entity type verification (mission-extractor.ts)

Find the Pass 3 system prompt in `mission-extractor.ts`. Add this instruction to the key_discriminator generation:

```
CRITICAL: The key_discriminator must verify TWO things:
1. ENTITY TYPE тАФ Is this business actually the type the user asked for? Google Places often returns related-but-wrong business types. A brewery is NOT a pub unless it operates a public bar. A barber is NOT a hairdresser. A beauty salon is NOT a hairdresser unless it offers hair cutting/styling. A restaurant is NOT a pub unless it has a bar area open to non-diners.
2. CONSTRAINT SATISFACTION тАФ Does the business meet the user's specific requirements?

Both must be true for a result to be valid.

In the key_discriminator text, explicitly state what the entity type IS and what it IS NOT. Examples:
- "pubs" тЖТ "Is a pub тАФ an establishment primarily serving drinks with a bar area open to walk-in customers. NOT a brewery production facility, NOT a restaurant without a bar, NOT a wine bar, NOT a members-only club."
```

2. Log the mismatch: `[ENTITY_TYPE_MISMATCH] "${leadName}" is actually a ${actual_type}, not a ${entityType}`
3. Include the mismatch in the lead's verification record so the Tower can see it

In the Layer 2 response handling (after parsing the JSON), add:

## CRITICAL RULES

1. Entity type verification is IN ADDITION TO constraint verification тАФ both must pass
2. The entityType parameter is optional тАФ if not provided, skip entity type checking (backwards compatible)
3. Don't be too strict тАФ a brewery WITH a public taproom IS a valid pub. A beauty salon that ALSO does hair IS a valid hairdresser. The judge should use common sense.
4. The entity_type_mismatch field should flow through to the delivery summary so the user can see WHY certain candidates were excluded
5. Do NOT change Google Places queries тАФ the fix is in verification, not discovery
6. This is the same philosophy as the constraint verification: the LLM reads the website and judges. No hardcoded lists of "what is a pub" тАФ the LLM knows.

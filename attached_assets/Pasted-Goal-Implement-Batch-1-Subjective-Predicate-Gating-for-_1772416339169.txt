Goal: Implement Batch 1 Subjective Predicate Gating for Layer 2 constraints.

Problem:
Queries like "Find nice bars in Manchester" or "Find good pubs in Leeds" currently RUN immediately. They must CLARIFY what the subjective word means, unless the user already gave measurable criteria.

Target behavior:
If the user uses subjective adjectives (nice, good, best, great, amazing, decent, cool, vibes, lovely, popular, trendy, fancy, high-end, top, recommended, quality, etc) and has not provided measurable criteria, block execution and ask a short clarifying question to convert the subjective term into measurable filters.

Measurable criteria examples (should allow RUN without clarification):
- "lively", "quiet", "family friendly", "dog friendly", "late night", "open late", "live music", "beer garden", "sports", "rooftop", "cheap", "cocktail bar", "micropub", "has food", "good Guinness" (treat as measurable), "near X", "within 2 miles", "4.5 stars" etc.
If you are unsure, err toward CLARIFY not RUN.

Scope:
This gating must apply in BOTH places:
1) When deciding chat mode for a fresh user message (before creating a run)
2) When inside clarify_for_run session handling follow ups (so "nice" cannot slip through as refinement)

Implementation requirements:
1) Add a new constraint type in the ConstraintContract: subjective_predicate
Fields:
- label: the raw term(s) detected, eg "nice"
- hardness: default soft
- verifiability: proxy (because it is not directly verifiable)
- required_inputs_missing: include "subjective_definition" when unclear
- can_execute false when missing definition
- why_blocked: "Your request includes a subjective term ('nice'). Tell me what you mean so I can search accurately."
- suggested_rephrase: produce a 1 line example rewrite using measurable choices

2) Detection:
Create a function detectSubjectiveTerms(text) that returns list of matched subjective tokens.
Create a function hasMeasurableCriteria(text) that detects any measurable criteria. It can be a regex list, keep it simple.

Logic:
If detectSubjectiveTerms returns non-empty AND hasMeasurableCriteria is false:
- return CLARIFY_FOR_RUN with missing field "subjective_definition"
- ask: "When you say 'nice', what do you mean? Pick one: cosy and quiet, lively and busy, trendy, upscale, cheap, good beer, good food, live music, or tell me your own criteria."
- Do not start tools.
- Keep location handling as-is (location may still be missing). If location is missing too, ask both, but keep it tidy: ask location first, then subjective definition, in the same clarify response.

3) Do not explode scope:
Do not build a taxonomy. Just a short list of options.
No refactor of router. Minimal change.

4) Tests:
Add regression tests that cover:
A) "Find nice bars in Manchester" -> CLARIFY, asks what "nice" means, no RUN
B) "Find good pubs in Leeds" -> CLARIFY
C) "Find lively bars in Manchester" -> RUN (no clarify)
D) "Find nice pubs in Bristol with live music" -> should still CLARIFY for "nice" OR treat live music as measurable and still CLARIFY for "nice" (must not RUN immediately)
E) "Find nice pubs in Bristol that opened in last 12 months" -> CLARIFY includes both: subjective definition + time proxy choice (compound constraints)

Deliverables in your response:
- Exact files changed
- New helper functions added
- Test file(s) updated and test command
- Brief note on detection lists so we can tune later

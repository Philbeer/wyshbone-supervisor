## Task: Add GPT-4o web search as a per-candidate verification fallback when website visits fail

### Context
During constraint verification, the agent visits each candidate's website to check for specific evidence (e.g. "live music mentioned on website"). Many sites block the agent with bot protection, Cloudflare challenges, or are simply down. These candidates end up as "unverified" with snippet-only evidence, even though the business is real and probably meets the constraint.

Currently these blocked candidates get marked as unverified and we move on. Instead, we should fall back to GPT-4o web search to verify that specific candidate.

### How it should work

1. During the website verification step, after attempting to visit a candidate's website:
   - If the visit SUCCEEDED → use the evidence found, no fallback needed
   - If the visit FAILED (bot-blocked, timeout, site down, no usable content) → trigger GPT-4o verification fallback for this specific candidate

2. The fallback is a focused, single-candidate verification call:
   - Call GPT-4o with web search enabled
   - Prompt: "Search for [candidate_name] in [candidate_location]. Find their website or any authoritative source. Determine whether [constraint_description] is genuinely true for this specific business. Provide the evidence you found and the source URL."
   - This is NOT a discovery search — we already know the candidate exists. We just need to verify the constraint.

3. Parse the GPT-4o response:
   - If evidence found → mark candidate as verified, source_tier = "web_search_fallback"
   - If no evidence found → mark as unverified (same as before)
   - If GPT-4o confirms the constraint is NOT met → mark as CONTRADICTED

4. Collect all candidates that needed fallback verification and process them as a batch at the end of the verification step (not inline per-candidate) to keep the pipeline clean:
   - After all direct website visits complete, gather the list of failed/blocked candidates
   - Run GPT-4o verification on each one sequentially
   - Merge the fallback results back into the candidate list before Tower evaluation

5. Log clearly:
   - "Website visit failed for [candidate_name]: [reason]. Queued for GPT-4o verification fallback."
   - "GPT-4o fallback: Verifying [N] blocked candidates..."
   - "GPT-4o fallback result for [candidate_name]: [verified/unverified/contradicted] — [evidence summary]"
   - "Verification complete: [X] direct, [Y] via fallback, [Z] still unverified"

6. Mark the evidence source on each candidate so Tower and BJ can see how it was verified:
   - source_tier: "first_party" (direct website visit worked)
   - source_tier: "web_search_fallback" (GPT-4o verified after direct visit failed)
   - source_tier: "snippet" (both direct visit and fallback failed or were not attempted)

### Artefact and Tower handling
The fallback evidence is NOT a separate artefact. It gets attached to the same candidate's verification record, replacing the failed direct visit evidence. The source_tier field distinguishes how it was obtained. Tower judges the evidence on its merits regardless of source_tier — a "live music every Saturday" quote is equally valid whether it came from a direct website visit or a GPT-4o web search. Tower does not need any code changes for this.

### Important
- This is a VERIFICATION fallback, not a discovery mechanism — we are NOT finding new candidates here
- Only trigger the fallback when the direct website visit fails — don't use it as a shortcut
- Rate limit GPT-4o calls — process the fallback batch sequentially
- The fallback results feed into the existing Tower and BJ pipeline unchanged

### Test expectation
Run B06 (Arundel live music). Previously many pubs show as "unverified / snippet" because their websites blocked the agent. After this change, those should get GPT-4o fallback verification. More pubs should end up as "verified" with web_search_fallback source tier. The number of "snippet only" results should drop significantly.

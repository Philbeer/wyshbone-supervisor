IMPLEMENTATION TASK — GPT-4o Primary Search Execution Path

CONTEXT: We are adding a new search mode to Wyshbone. Currently all queries go through: Intent Extraction → Mission Planning → Google Places discovery → website visits → evidence extraction → per-candidate Tower verification → final Tower delivery judgement. We're adding an alternative execution path where GPT-4o web search handles discovery AND verification in a single call, dramatically simplifying the pipeline for certain query types.

The UI is being updated in parallel to send a new field `execution_path: "gp_cascade" | "gpt4o_primary"` in the query payload.

IMPORTANT: This is ADDITIVE. Do not remove, rename, or reorganise any existing code. All current functionality must continue to work identically when execution_path is "gp_cascade" or absent/undefined (treat missing as "gp_cascade" for backward compatibility).

WHAT TO BUILD:

1. ACCEPT execution_path FROM THE UI
- Find where the incoming query payload is received from the UI (the API route that handles new user queries).
- Add `execution_path` as an optional field. If absent or "gp_cascade", everything works exactly as it does now — no behaviour change at all.
- Pass `execution_path` through to wherever mission execution is orchestrated (likely `mission-executor.ts` based on the codebase).

2. KEEP INTENT EXTRACTION AND PLANNING UNCHANGED
- The intent extractor (`intent-extractor.ts`) runs identically regardless of execution_path. It still determines mission_type, extracts constraints, produces the Pass 1/2/3 outputs, etc.
- Mission planning still runs. The query_class / mission_type is still determined.
- The trust card and mission understanding chat bubbles still get emitted exactly as they do now.
- DO NOT modify intent-extractor.ts or any planning logic.

3. ADD gpt4o_primary EXECUTION BRANCH IN MISSION EXECUTOR
- In the mission executor (likely `mission-executor.ts`), add a conditional branch: if `execution_path === "gpt4o_primary"`, execute the new path instead of the existing GP + cascade pipeline.
- The existing GP + cascade code must remain completely untouched inside its own branch.

THE NEW gpt4o_primary EXECUTION PATH:

Step A — Emit status chat bubble:
- Send a chat bubble to the UI saying something like: "Searching with GPT-4o web search for: [brief description of what we're looking for based on the intent extraction output]"
- Use the same chat bubble / status emission mechanism that the existing pipeline uses for "Searching Google Places..." messages.

Step B — Call GPT-4o with web search:
- Use the OpenAI API (the project already has the OpenAI SDK configured for gpt-4o-mini calls — use the same client/configuration but with model "gpt-4o" and web search enabled).
- To enable web search, use the OpenAI Responses API:
```typescript
  const response = await openai.responses.create({
    model: "gpt-4o",
    tools: [{ type: "web_search_preview" }],
    input: promptText
  });
```
  If the Responses API is not available in the current SDK version, use Chat Completions with the web search tool:
```typescript
  const response = await openai.chat.completions.create({
    model: "gpt-4o",
    tools: [{ type: "web_search_preview" }],
    messages: [{ role: "user", content: promptText }]
  });
```
  Check what OpenAI SDK version is installed and use whichever approach works. The key requirement is: GPT-4o with web search enabled.

- The prompt should be structured like this (adapt the actual constraint/entity details from the intent extraction output):
You are a research assistant finding specific entities. Search the web thoroughly.
TASK: Find [entity_description from Pass 3] that [constraints from Pass 2].
Location: [location from intent extraction]
For EACH result you find, provide:

name: The entity/business name
description: Brief description of what they do
evidence: The specific evidence that they match the search criteria (quote or paraphrase from your source)
source_url: The URL where you found this information
location: Their address or location if available
confidence: "high" if evidence is direct and clear, "medium" if inferred or from secondary source, "low" if uncertain

Return results as a JSON array. Be thorough — search multiple angles if needed. Only include results where you found genuine evidence. Do not fabricate or assume.
Respond with ONLY a JSON object in this format:
{
"results": [
{
"name": "...",
"description": "...",
"evidence": "...",
"source_url": "...",
"location": "...",
"confidence": "high|medium|low"
}
],
"search_summary": "Brief description of what you searched for and how many results you found",
"coverage_assessment": "How comprehensive do you think these results are? Are there likely more to find?"
}

Step C — Parse GPT-4o response into existing leads format:
- Parse the JSON response from GPT-4o.
- Convert each result into the same lead object format that the existing GP pipeline produces (look at what `accumulateLeads()` creates — match that structure).
- Each lead should have:
  - A generated unique ID (use the same ID generation pattern as existing leads)
  - The name, location/address fields mapped from the GPT-4o response
  - source_tier set to "gpt4o_web_search"
  - The evidence text stored so it can be included in the delivery
- Handle JSON parsing errors gracefully — if GPT-4o returns malformed JSON, log the error and try to extract what we can, or report an honest failure.

Step D — Agent completeness check (optional second search):
- After parsing results, evaluate: did we get enough results?
- If the result count is low (e.g., fewer than 5) AND the coverage_assessment suggests there might be more, do ONE additional GPT-4o web search with a different search angle.
  - Example: if first search was "organisations that work with Blackpool Council", second might be "Blackpool Council partners contractors suppliers"
  - Deduplicate results across searches by name (fuzzy match is fine — exact string match on lowercased trimmed names is sufficient for now)
- Cap at maximum 3 search rounds total to control cost.
- Emit a status chat bubble for each additional search: "Searching from a different angle..."

Step E — Create step artefact:
- Create the same kind of step artefact that the existing pipeline creates after GP search results come in.
- The artefact should contain: the leads found, the search method used ("gpt4o_web_search"), the number of searches performed, and the raw evidence.
- Use the existing artefact creation functions/patterns — look at how the GP search step creates its artefact and follow the same pattern.
- This is important: we are keeping the step artefact architecture even though we're not doing per-candidate Tower verification. The artefact records what happened at this step.

Step F — Skip per-candidate Tower verification:
- Do NOT call Tower's per-candidate semantic-verify endpoint for gpt4o_primary results.
- The reasoning: GPT-4o web search already found and verified these candidates in the same call — the search result IS the evidence.
- BUT: do not remove or modify the existing per-candidate Tower verification code. It must continue to work for gp_cascade mode. Simply don't invoke it in the gpt4o_primary branch.
- For the attribute_verification artefact: still create it, but populate it with source_tier = "gpt4o_web_search" and the evidence directly from the GPT-4o response. The verification status can be set to "verified" with confidence based on what GPT-4o reported.

Step G — Final Tower delivery judgement:
- DO call Tower's final delivery judgement endpoint, exactly as the existing pipeline does.
- Send the complete results package to Tower for holistic quality assessment.
- Tower will check: did we answer the right question? Right location? Enough results?
- Handle Tower's verdict (ACCEPT, ACCEPT_WITH_UNVERIFIED, CHANGE_PLAN, STOP) the same way the existing pipeline does.

Step H — Deliver results:
- Use the existing `run-receipt.ts` / `emitRunReceipt()` pattern to assemble and deliver the final results.
- Results should appear in the UI exactly as existing results do — same card format, same fields.
- The only visible difference should be the source_tier showing "gpt4o_web_search" instead of "google_places" or "brave_search".

CONTEXT
Both executor adapters set entity.verified at the wrong granularity:

- gp-cascade-adapter.ts:41 sets verified to the WHOLE LOOP's Tower verdict
  (result.towerVerdict). If that loop's Tower said fail (e.g. live music
  constraint missing on most GP results), ALL 14 entities get verified=false,
  even leads where the per-lead match_valid is true.

- gpt4o-adapter.ts:49 hardcodes verified: true for every entity, regardless
  of whether that specific lead's match_valid is true.

This breaks loop-skeleton.ts line 749's combinedLeads.filter(l =>
l.verified === true), causing:
- Wrong artefact title ("Combined delivery: 1 leads from 2 loops" when 15
  were delivered)
- Wrong Tower input when the combined call fires (rare now that yesterday's
  skip logic is in place, but still a latent bug)

THE TRUTH PER LEAD already exists at result.deliverySummary.delivered_exact[i].match_valid.
mission-executor.ts:2720 sets it per-lead based on actual per-lead semantic
verification. We just need both adapters to read it.

REPLACE with:

  const entities: ExecutorEntity[] = result.leads.map(lead => {
    const dsLeads = (result.deliverySummary as any)?.delivered_exact || [];
    const dsLead = dsLeads.find((dl: any) => dl.name === lead.name);
    const matchEvidence = dsLead?.match_evidence || dsLead?.supporting_evidence || [];

    // Per-lead verified flag — source of truth is dsLead.match_valid which
    // mission-executor.ts sets based on per-lead semantic verification.
    // Fall back to whole-loop Tower verdict only when match_valid is undefined
    // (preserves current behaviour for runs without per-lead data).
    const wholeLoopPassed = result.towerVerdict === 'pass' || result.towerVerdict === 'accept';
    const verifiedFlag = dsLead?.match_valid === true
      ? true
      : dsLead?.match_valid === false
        ? false
        : wholeLoopPassed;

    return {
      name: lead.name,
      address: lead.address,
      phone: lead.phone,
      website: lead.website,
      placeId: lead.placeId,
      source: 'gp_cascade',
      verified: verifiedFlag,
      verificationStatus: dsLead?.match_valid === true ? 'verified' : (result.towerVerdict ?? 'unknown'),
      evidence: matchEvidence,
    };
  });

  const entities: ExecutorEntity[] = result.leads.map(lead => {
    const dsLeads = (result.deliverySummary as any)?.delivered_exact || [];

KNOWN LIMITATION (not fixed here)
The gpt4o-search.ts source itself hardcodes match_valid: true for every
lead. So for runs that go purely through the gpt4o_search executor with no
real per-lead verification, every entity still gets verified=true regardless
of evidence quality. Fixing this means changing gpt4o-search.ts to compute
match_valid per-lead from its own verification results — bigger change,
belongs in v1 hardening pass (item 9) if the gpt4o path needs honest
per-lead truth.

CONTEXT
The combined Tower judge receives constraints two ways:
  1. Via successCriteria.hard_constraints (labels like 'attribute_check')
     and successCriteria.structured_constraints (full payloads with values)
  2. Via the artefact payload itself (the thing the LLM judge primarily reads)

The combined_delivery artefact payload currently includes leads, counts,
and per-loop info, but does NOT include the constraint values. So when the
LLM Tower judge looks at the artefact, it sees no constraint values inside
the payload and produces a misleading narrative like:

  "[Tower] pass: Delivered 10 pubs in Arundel with no hard constraints to verify."

— even when the query has a clear hard constraint like "live music" that
per-lead semantic verification has already verified.

The user-facing 4-bullet message is still honest (it reads from
delivery_summary, not from the combined Tower rationale). But the activity
feed shows the misleading rationale, which is confusing for diagnostic and
demo purposes.

GOAL
Include the mission's constraints in the combined_delivery artefact payload
so the LLM Tower judge sees them inside the artefact it judges, and can
produce accurate rationale text.

        total_loops: totalLoops,
        executors_tried: executorsTriedSoFar,
        requested_count: requestedCount,
        delivered_count: deliveredLeads.length,

        delivery_note: deliveryNote,

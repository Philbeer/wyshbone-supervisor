Purpose:
Fix semantic dishonesty by enforcing correct interpretation and labelling when the user intent includes a relationship predicate.

DO NOT add tools. DO NOT change discovery. DO NOT change UI components directly.
Interpretation and delivery semantics only.

🔧 SUPERVISOR PROMPT (paste into Replit)

Context
Wyshbone currently delivers results that are semantically incorrect when the user intent requires a relationship (e.g. “works with councils”).
The system is confidently wrong because it treats word association as intent satisfaction.

This task is to fix interpretation, labelling, and delivery honesty in the Supervisor layer only.

Objective

Prevent Wyshbone from claiming “matches” when the user intent requires a relationship that has not been evidenced.

Required Behaviour Changes
1. Detect relationship predicates in user intent

In the intent parsing / normalisation phase, detect relationship predicates including (but not limited to):

“works with”

“supplies”

“serves”

“partners with”

“supports”

“clients of”

“vendors to”

If any are present, set a run-level flag:

requires_relationship_evidence = true

This flag must be persisted in the run context and passed to downstream artefacts.

2. Candidate vs Match enforcement

When requires_relationship_evidence = true:

Google Places results are candidates only

NO result may be labelled:

“match”

“verified match”

“meets criteria”

Until relationship evidence exists

Word-level association (e.g. name contains “Council”) MUST NOT be treated as intent satisfaction.

3. Relationship evidence definition

A result may only be upgraded from candidate to match if at least one of the following exists:

Website language explicitly stating the relationship
(e.g. “we work with local councils”, “clients include councils”)

Case studies, partners, or clients naming councils

Credible third-party confirmation

If none exist:

relationship_verified = false
4. Delivery summary honesty

If requires_relationship_evidence = true AND verified_relationship_count = 0:

❌ Forbidden language:

“I found X results that match”

“These organisations meet your criteria”

✅ Required language (or equivalent):

“I found organisations associated with councils, but could not verify that they work with councils.”

“No relationship evidence could be confirmed.”

Supervisor must ensure no UI-visible artefact contradicts this.

5. “Unverified” semantics correction

“Unverified” must mean:

Verification was attempted and failed.

NOT:

Verification not yet done.

If relationship verification fails, label explicitly as:

“Relationship not verified”

“No evidence found”

6. Output contract to Tower

Supervisor must pass to Tower:

requires_relationship_evidence: boolean
verified_relationship_count: number

Supervisor MUST NOT override Tower verdicts.
Supervisor MUST NOT claim success when verification failed.

Acceptance Test

Query:

“organisations that work with councils”

Result:

“County Councils Network”

Expected Supervisor behaviour:

Treated as candidate only

NOT a match

Delivery summary explains relationship could not be verified

No “X results that match” language

Goal

Make it impossible for Wyshbone to be confidently wrong about relationships.

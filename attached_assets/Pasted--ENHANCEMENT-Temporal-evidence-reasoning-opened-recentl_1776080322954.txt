# ENHANCEMENT: Temporal evidence reasoning — "opened recently" and time constraints

## REPO: wyshbone-supervisor-Post-CC

## CONTEXT

When a user searches for "micro pubs that opened recently in West Sussex", the system creates a time_constraint: `{ type: "time_constraint", field: "opening_date", operator: "within_last", value: "recent", hardness: "hard" }`. This routes through the website evidence pipeline correctly.

But the evidence judge searches for literal phrases like "opened recently" and finds nothing — because businesses don't write that on their website. A human would infer recency from signals: "Now Open!", "Grand Opening", "est. 2025", a modern website template, Google reviews only from the last few months.

This is the same pattern as the inferential reasoning we added for "independent" — the LLM needs to be taught what temporal signals look like, and the evidence judge needs permission to accept them.

### Change 5: Google Places review date as evidence (mission-executor.ts)

This is a NEW evidence source that doesn't require website visits. Google Places returns review data including timestamps. If the earliest review for a business is from the last 6-12 months, that's strong evidence the business is new.

In the evidence verification loop in `mission-executor.ts`, when processing a lead with a time_constraint, add a check BEFORE the website evidence pipeline:

```typescript
// Check Google Places review dates for temporal evidence
if (timeConstraint && lead.reviews && Array.isArray(lead.reviews)) {
  const reviewDates = lead.reviews
    .map(r => new Date(r.time * 1000 || r.date || r.created_at))
    .filter(d => !isNaN(d.getTime()))
    .sort((a, b) => a.getTime() - b.getTime());
  
  if (reviewDates.length > 0) {
    const earliestReview = reviewDates[0];
    const monthsAgo = (Date.now() - earliestReview.getTime()) / (1000 * 60 * 60 * 24 * 30);
    
    if (timeConstraint.operator === 'within_last' && monthsAgo <= 12) {
      console.log(`[TIME_EVIDENCE] "${lead.name}" — earliest Google review is ${Math.round(monthsAgo)} months old — consistent with "opened recently"`);
      // Add as evidence item
      lead._temporal_evidence = {
        source: 'google_reviews',
        earliest_review: earliestReview.toISOString(),
        months_ago: Math.round(monthsAgo),
        signal: `Earliest Google review is from ${Math.round(monthsAgo)} months ago, suggesting the business opened around that time.`,
      };
    }
  }
}
```

Note: Check whether Google Places API data includes review timestamps in the format the system receives. If reviews are just text without dates, skip this step — the website evidence pipeline handles it instead.

### Change 6: Combine temporal evidence sources

For a lead being checked against a time constraint, the verification should combine:
1. **Google review dates** (if available) — strong structured signal
2. **Website content signals** — "now open", "est. 2025", modern design indicators
3. **Absence of historical content** — no old blog posts, no "celebrating 10 years" type content

The evidence judge should weigh these together. A lead with recent Google reviews AND "now open" on the website is high-confidence recent. A lead with only "modern-looking website" is low-confidence. Neither alone may be enough, but together they're strong.

Add this to the evidence judge prompt for time constraints:

```
For time constraints, combine multiple signals:
- Strong: explicit date ("est. 2025"), "now open" messaging, Google reviews only from last 12 months
- Medium: modern website, no historical content, "under new management"
- Weak: no clear signals either way

A business with 2+ medium signals or 1+ strong signal satisfies "opened recently."
A business with Google reviews going back several years does NOT satisfy "opened recently."
```

## CRITICAL RULES

1. Temporal verification is ADDITIVE — it doesn't replace existing verification, it adds temporal evidence on top
2. If no temporal signals are found, the lead fails the time_constraint (this is correct behaviour — the user asked for recent openings)
3. Do NOT hardcode specific years — the LLM should calculate recency relative to the current date
4. Google review dates are a SUPPLEMENT to website evidence, not a replacement — use both when available
5. This uses the same evidence pipeline as everything else — no new architecture needed

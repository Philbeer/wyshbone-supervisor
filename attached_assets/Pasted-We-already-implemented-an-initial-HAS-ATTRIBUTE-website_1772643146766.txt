We already implemented an initial HAS_ATTRIBUTE website matcher (currently biased to “live music” style keywords). Now refactor it into a generic attribute verification engine so ANY user attribute phrase works (eg “beer festivals”, “quiz nights”, “darts”, “function room”).

Goal:
Stop curve fitting. Default behaviour must work for arbitrary HAS_ATTRIBUTE strings without adding new per-attribute code.

Hard constraints:

Keep using existing WEB_VISIT artefact text and URLs. No new external calls.

Do not reduce honesty: “yes” still requires evidence snippet + source URL.

Prefer UNKNOWN over NO if not found.

Keep current artefact type and payload shape stable if possible, but add fields if needed.

Required changes:

Generic matcher:

Treat each HAS_ATTRIBUTE constraint value as a raw phrase attrRaw (string).

Generate a set of match variants automatically from attrRaw:

lowercase

trim/punctuation stripped

hyphen/space normalization (“beer-festival” vs “beer festival”)

basic singular/plural variants for the last word (festival/festivals, band/bands) using simple rules:

if endsWith("s") -> also try without trailing "s"

else -> also try + "s"

optional: handle “&” vs “and”

Search the website text (case-insensitive) for any variant.

If found: status="yes" and store snippet + url + matched_variant

If not found: status="unknown" with reason="no_match_in_visited_pages"

Preserve the existing curve fitted matcher as an OPTIONAL overlay:

Keep a small synonym map (eg live music -> gig/open mic) but make it additive only.

The generic matcher must run even if no synonym map entry exists.

Only use synonym expansion if it increases recall; never force “yes” without direct text match.

Evidence quality:

Snippet should be short and readable (eg 160–240 chars around the match).

Save matched_variant in evidence to aid debugging.

CVL integration:

CVL HAS_ATTRIBUTE must refer to the generic attributeEvidence for that exact attrRaw string.

If synonym expansion is used, evidence should still be attached under the original attrRaw key, but include matched_variant showing what actually matched.

Artefact:

attribute_verification artefact must now include:

attribute_raw

matched_variant (when yes)

status and reason

snippet + url

Acceptance tests:
A) “Find pubs in Arundel that say they have live music on their website”

Should still work and produce yes/unknown with evidence.

B) “Find pubs in Arundel that mention beer festivals on their website”

Must produce attribute_verification where attribute_raw="beer festivals"

Must mark yes if any visited page contains “beer festival” or “beer festivals” (variant match)

Others unknown, not no.

C) A totally arbitrary attribute:

“Find pubs in Arundel that mention ‘sourdough pizza’ on their website”

System should not crash; attribute_verification must run and return unknown for most/all.

Output:

files changed

show the generic variant generation function

show where CVL reads attributeEvidence now

include example attribute_verification payload for test B

Non-goals:

Do not implement category specific heuristics beyond the optional synonym overlay.

Do not add extra page crawling yet (events pages etc). Just make matching generic first.

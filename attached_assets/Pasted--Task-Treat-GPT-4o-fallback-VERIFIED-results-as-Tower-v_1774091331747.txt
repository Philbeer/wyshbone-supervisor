## Task: Treat GPT-4o fallback VERIFIED results as Tower-verified

In server/supervisor/mission-executor.ts, the GPT-4o web search fallback section (search for GPT4O_FALLBACK in the file) runs when a lead's website was bot-blocked or Layer 1 found no evidence. When GPT-4o searches the web and returns "VERIFIED: [evidence]", the code currently sets:

er.evidenceFound = true;
er.evidenceStrength = 'weak';

This means Tower verdict stays null and the lead shows as "unverified" in the UI, even though GPT-4o with web search already confirmed the constraint with real sources. There's no point calling Tower again — GPT-4o's web search verification is at least as authoritative.

### The fix

Find the block inside the GPT-4o fallback where a verified result is processed (inside the `if (!hasContradiction)` block, after the contradiction check). It currently reads:
er.evidenceFound = true;
er.evidenceStrength = 'weak';
er.snippets = [fbContent.substring(0, 500)];
if (fbSourceUrl) er.sourceUrl = fbSourceUrl;
fallbackVerified++;

Change it to:
er.evidenceFound = true;
er.evidenceStrength = 'strong';
er.towerStatus = 'verified' as any;
er.towerConfidence = 0.75;
er.towerReasoning = 'Verified via GPT-4o web search fallback (website was bot-blocked or had no extractable evidence)';
er.snippets = [fbContent.substring(0, 500)];
if (fbSourceUrl) er.sourceUrl = fbSourceUrl;
fallbackVerified++;

Three changes:
1. evidenceStrength changes from 'weak' to 'strong' — GPT-4o confirmed it with sources
2. towerStatus set to 'verified' — so the UI shows "verified" instead of "unverified"
3. towerConfidence set to 0.75 and towerReasoning explains the source — for the AFR audit trail

This means bot-blocked leads like The White Swan where GPT-4o found "clearly features live music based on multiple authoritative sources" will show as verified, which they genuinely are.

No other changes needed.

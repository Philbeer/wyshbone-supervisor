Implement real verification for HAS_ATTRIBUTE constraints (starting with live music) by extracting attribute evidence from the existing website text that WEB_VISIT already fetched.

Do not change UI. Do not add new external APIs. Do not re-fetch websites if we already have WEB_VISIT text for the lead.

Goal:
A query like “Find pubs in Arundel that say they have live music on their website” must no longer behave identically to “Find pubs in Arundel”.
It must attempt website-based attribute verification for each lead with a website, and produce evidence-backed results.

Hard constraints:

Use existing run artefacts, especially WEB_VISIT (or whatever artefact contains visited page text + final URL).

No new API calls. No new scraping.

YES requires evidence: a snippet and the URL it came from. Otherwise UNKNOWN.

Prefer UNKNOWN over NO unless you have explicit negative evidence.

Implementation requirements:
A) Add an extraction step that runs after WEB_VISIT and before delivery summary:

Inputs:

extracted constraints (HAS_ATTRIBUTE values)

the set of leads delivered/candidates including place_id + website

the existing WEB_VISIT artefacts for this run (page text + url)

Output:
attributeEvidence[placeId][attribute] = { status: "yes"|"unknown"|"no", snippet?: string, url?: string }

B) Implement matcher for live music (minimal first):

keywords: "live music", "live band", "live bands", "music night", "gig", "open mic"

matching should be case-insensitive

snippet should be a short excerpt around the match (eg 120–200 chars)

C) Emit a new artefact:

type: attribute_verification

payload includes per lead:

place_id, name, website

per attribute: status, snippet, url

plus summary counts: checked_sites_count, yes_count, unknown_count

D) Update CVL HAS_ATTRIBUTE verification:

It must consult attributeEvidence for each lead.

Hard HAS_ATTRIBUTE:

yes => can be exact

unknown/no => not exact (closest), with a clear reason string

E) Update delivery summary generation:

Leads should be marked exact only when hard HAS_ATTRIBUTE is verified yes.

Otherwise closest with violation reason "attribute not verified" or "no evidence found".

F) Special-case: if user text contains “on their website”, treat that as explicit verification method and do NOT trigger proxy clarification for live music.

Outputs required:

files changed

where WEB_VISIT artefacts are read

sample attribute_verification artefact JSON

demonstrate with a test run:
“Find pubs in Arundel that say they have live music on their website”
and show that the attribute_verification artefact has some yes/unknown results.

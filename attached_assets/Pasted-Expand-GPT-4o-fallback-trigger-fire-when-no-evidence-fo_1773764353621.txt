Expand GPT-4o fallback trigger: fire when no evidence found, not just when bot-blocked
Currently the GPT-4o verification fallback only fires when a website visit is bot-blocked. But there's a gap: when the website loads fine and the crawler finds no evidence (because the evidence is on a blog post, Facebook, Google snippet, or a page the crawler didn't reach), the candidate just gets marked "no evidence" and is dropped.
Change the fallback trigger condition so that GPT-4o fires for ANY candidate where the website visit produced no evidence for the constraint, not just bot-blocked candidates. Specifically:
Current logic (in the fallback batch section after all website visits complete):

Only candidates where isBotBlocked && !evidenceFound get queued for GPT-4o fallback

New logic:

Any candidate where !evidenceFound gets queued for GPT-4o fallback, regardless of whether the website visit succeeded or was blocked

This means:

Bot-blocked + no evidence → GPT-4o fallback (same as before)
Website visited successfully + no evidence found → GPT-4o fallback (NEW)
Website visited successfully + evidence found → no fallback needed (same as before)

Update the logging to distinguish the two cases:

"Website visit failed for [name]: [reason]. Queued for GPT-4o verification fallback."
"Website visit succeeded for [name] but no evidence found. Queued for GPT-4o verification fallback."

Keep the source_tier tagging:

If the direct visit found nothing and GPT-4o finds evidence: source_tier: "web_search_fallback"
If the direct visit found evidence: source_tier: "first_party" (unchanged)

Test expectation: Run B06 (Arundel live music). The George at Burpham's website was visited successfully (5 pages) but no live music evidence was extracted. With this change, GPT-4o should now search for "George at Burpham live music" and find it — there's a blog post and Google snippet that clearly mention live music events. This should increase the number of verified pubs delivered.
Don't change anything else — the cascade, the Tower pipeline, and the evidence extraction are all fine. Just widen the fallback trigger.

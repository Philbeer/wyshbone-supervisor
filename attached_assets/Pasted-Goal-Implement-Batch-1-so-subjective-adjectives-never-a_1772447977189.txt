Goal: Implement Batch 1 so subjective adjectives never auto-execute. They must trigger the same CLARIFY gate pattern used for time predicates. Do not refactor the overall flow.

Non negotiables

If a subjective term is present and not defined in measurable terms, the run must not execute.

Behaviour must mirror the time-predicate gate (same structure, same contract fields, same user experience quality).

Measurable adjectives must still execute normally.

What to implement
1) Subjective predicate detection (new extractor)

Detect subjective terms in the user request, including at least:

Subjective list (gate these):

nice, good, best, great, amazing, cool, popular, decent, vibes, vibey, top, favourite, trendy

Add common variants:

nicer, nicest, better, greatest, coolest, most popular, “best rated”, “best places”, “good atmosphere”

Output a constraint item for each subjective term found:

type = "subjective"

hardness = "soft" (default)

verifiability = "unverifiable"

can_execute = false unless defined by measurable criteria (see below)

why_blocked = "This request uses a subjective term that needs clarification"

required_inputs_missing = ["definition_of_<term>"]

If multiple subjective terms exist, combine into one clarify gate, but preserve each missing definition in required_inputs_missing.

2) Explicit measurable allow-list (do NOT gate these)

Maintain an allow-list of measurable terms that should not trigger this gate, for example:

lively

quiet

late-night / open late

family-friendly

live music

dog friendly

sports / football on TV

rooftop

beer garden

cocktails

food served

wheelchair accessible

cheap / budget (price-ish but still actionable)

“near X” / “in city centre” / radius constraints

Important: allow-list must override subjective detection when both appear in the same phrase, for example:

“nice lively bars” still gates because “nice” is unresolved, but “lively” stays as a usable measurable option.

3) Define when a subjective term becomes “defined” and executable

A subjective term can be considered “defined” only if the user provides measurable criteria that replaces it, such as selecting from suggested options or explicitly stating criteria.

Implement this as:

The clarify response must request a measurable definition.

When the user replies with measurable criteria (one or more allow-list items), the system should:

remove or mark the subjective constraint as resolved

add the measurable constraints instead (or map it as the definition payload)

flip constraint_contract.can_execute = true and proceed

Do not silently interpret “nice” as “popular” or “highly rated”. It must be user-chosen.

4) Suggested rephrases and option set

Populate suggested_rephrase and clarification_options (or whatever your existing contract supports for the time gate) with measurable choices.

Minimum option buttons:

Lively

Quiet

Cosy

Late-night

Live music

Good for food

Beer garden

Dog friendly

Also include at least 2 suggested rephrases that preserve the user’s location:

“Find lively bars in Manchester city centre”

“Find cosy pubs in Manchester with live music”

5) Compound queries rule

If the query contains subjective term(s) plus other constraints (location, time, attributes):

Still CLARIFY first for the subjective term

Preserve the other constraints in the contract so they remain in effect after clarification

Example:
“Find nice pubs in Bristol that opened last year”

Must CLARIFY for “nice”

Must retain the time predicate constraint and location

6) Unit tests (required)

Add tests that assert the contract output, not just a high level status.

Cases:

“Find nice bars in Manchester” -> CLARIFY (can_execute false, type subjective, required_inputs_missing includes definition_of_nice)

“Find good pubs in Leeds” -> CLARIFY

“Find lively bars in Manchester” -> RUN (no subjective constraints produced)

“Find nice pubs in Bristol that opened last year” -> CLARIFY for nice, and still contains the time predicate constraint

“Find best bars in Soho” -> CLARIFY

“Find bars with good cocktails in Manchester” -> CLARIFY (good is subjective even if cocktails is measurable, unless user defines what “good” means)

7) Logging and safety

Add a single clear log line when this gate triggers, same style as time gate, so we can see it in AFR:

constraint_gate_triggered: subjective

Definition of done

Subjective terms never auto execute.

Measurable terms run.

Contract fields mirror time gate quality.

Tests pass.

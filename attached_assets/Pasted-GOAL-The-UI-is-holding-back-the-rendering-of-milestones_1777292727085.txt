GOAL
The UI is holding back the rendering of milestones (and possibly the chat response) until either an "intent_narrative" artefact arrives OR a 25-second safety timeout fires. For CHAT-route messages (e.g. wine questions, greetings, knowledge queries), no intent_narrative is ever generated by the supervisor, so the timeout always fires — making chat responses feel artificially slow.

The fix is small: when a CHAT route is detected, do not block on intent_narrative at all. Render immediately.

CHANGE — Drop the chat-route hold in /client/src/components/results/LiveActivityTicker.tsx

In LiveActivityTicker.tsx, find the dedicated intent_narrative poll useEffect (it starts with the comment "── Dedicated fast poll for intent_narrative" around line 600-620 onwards). It contains:

  // Safety: show milestones after 25 seconds even if intent never arrives
  const fallbackTimer = setTimeout(() => {
    if (!fetchedIntentNarrativeRef.current) {
      setIntentResolved(true);
    }
  }, 25000);

Change the timeout from 25000 to 1500. That's it. One number change.

The rationale: 25 seconds was chosen as a maximum-patience fallback when the system was thought to be slow. Now that backend chat returns in ~5 seconds, this 25s fallback is the dominant source of perceived latency for chat. Reducing it to 1.5s means chat responses show up almost immediately, while genuinely fast intent_narrative arrivals (for SEARCH routes) still pre-empt the fallback as before.

Do NOT change anything else. Do NOT remove the timer entirely (it's a safety net for SEARCH flows where the artefact might be slow to write). Do NOT modify the intent poll logic, the dispatch event, the milestones rendering, or anything else in this file.

Wire Up Cross-Run Learning (Roadmap 5.2)
Context
The learning layer infrastructure exists but is dead code — nothing calls it during actual runs. The Tower already generates learning_update in its judge-artefact response (confirmed in routes-judge-artefact.ts), but the Supervisor ignores it. The learning store tables exist in Supabase (migration 005). The query shape key functions exist (query-shape-key.ts). The learning store read/write functions exist (learning-store.ts). None of them are called.
This task wires everything together so the system actually learns from past runs.
What "learning" means concretely
After each run, the system records: "For queries shaped like find_venues::pubs::GB::attr:live_music, Google Places found 3 results and GPT-4o found 8. Next time, skip GP and start with GPT-4o."
Before each run, the planner checks: "Have I seen this query shape before? What worked last time?" and adjusts accordingly.

// Persist learning update if Tower returned one
if (judgement.learning_update && judgement.learning_update.query_shape_key) {
  const lu = judgement.learning_update;
  const updates: Record<string, unknown> = {};
  if (Array.isArray(lu.changed_fields)) {

Change 6: Planner reads learning store before choosing executor
In server/supervisor/reloop/planner.ts, modify rulesPlan to check learned data:
typescriptimport { readLearningStore } from '../learning-store';
import { computeQueryShapeKey, deriveQueryShapeFromGoal } from '../query-shape-key';

export async function rulesPlan(context: PlannerContext): Promise<PlannerDecision> {
  const { loopNumber, loopHistory, executionPath, availableExecutors, circuitBreaker } = context;
  const executorsTriedSoFar = new Set(loopHistory.map(r => r.plannerDecision.executorType));

  if (circuitBreaker) {
    const fastest = availableExecutors[0] ?? 'gp_cascade';
    return { executorType: fastest, reasoning: `Circuit breaker active — using ${fastest}.` };
  }

  // On first loop, check learning store for this query shape
  if (loopNumber === 1 && context.mission) {
    if (executionPath === 'gpt4o_primary') {
      return { executorType: 'gpt4o_search', reasoning: 'User explicitly requested GPT-4o primary path.' };
    }

    try {
      const shapeInput = deriveQueryShapeFromGoal({
        business_type: context.mission.businessType,
        location: context.mission.location,
        country: context.mission.country,
        constraints: context.constraints?.hardConstraints?.map(h => ({
          type: h, field: h, hard: true,
        })),
      });
      const shapeKey = computeQueryShapeKey(shapeInput);
      const learned = await readLearningStore(shapeKey);

      if (learned.exists) {
        // If previous runs for this shape needed multiple loops,
        // skip GP and go straight to GPT-4o
        if (learned.knobs.search_budget_pages >= 2 && availableExecutors.includes('gpt4o_search')) {
          const reason = `Learning store: previous runs for shape "${shapeKey}" needed ${learned.knobs.search_budget_pages} loops — skipping GP cascade, starting with GPT-4o.`;
          console.log(`[RELOOP_PLANNER] Loop ${loopNumber}: ${reason}`);
          return { executorType: 'gpt4o_search', reasoning: reason };
        }
        console.log(`[RELOOP_PLANNER] Loop ${loopNumber}: learned data for shape "${shapeKey}" — search_budget_pages=${learned.knobs.search_budget_pages}, proceeding with default GP cascade`);
      }
    } catch (learnErr: any) {
      console.warn(`[RELOOP_PLANNER] Learning store read failed (non-fatal): ${learnErr.message}`);
    }

    return { executorType: 'gp_cascade', reasoning: 'First loop — using GP cascade (cheap, fast, structured data).' };
  }

  // ... rest of existing logic unchanged ...

What this enables
After this change:

Every run computes a query shape key (e.g. find_venues::pubs::GB::attr:live_music)
The Tower evaluates whether to suggest policy changes based on the run outcome
The Supervisor persists those changes to the learning store
The Supervisor also records which executor found the most results
On the NEXT run with a similar query shape, the planner checks the learning store and starts with the best executor
The UI shows "Applied policy" with learned/default badges (already built)

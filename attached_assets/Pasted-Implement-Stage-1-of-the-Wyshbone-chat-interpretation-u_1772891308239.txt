Implement Stage 1 of the Wyshbone chat interpretation upgrade.

Goal

Replace brittle chat interpretation with a production-grade structured mission extractor that uses:

1. a fixed constraint taxonomy
2. a two-pass LLM extraction flow
3. deterministic schema validation
4. regex only for tiny cleanup, not language understanding

What Stage 1 must do

Given messy user chat, produce a structured mission object with:

- entity_category
- location_text
- constraints[]
- mission_mode

The system must stop preserving user surface phrasing and instead extract semantic meaning.

Canonical constraint taxonomy

Use only these constraint types:

- entity_discovery
- location_constraint
- text_compare
- attribute_check
- relationship_check
- numeric_range
- time_constraint
- status_check
- website_evidence
- contact_extraction
- ranking

Mission modes

Use only these mission modes:

- research_now
- monitor
- alert_on_change
- recurring_check

Two-pass LLM architecture

Pass 1: Meaning Extraction
Purpose:
Interpret what the user is actually asking for in clean semantic language.

Example input:
find pubs in arundel that have the word swan in the name

Example pass 1 output:
The user wants pubs in Arundel whose business name contains the word "swan". This is a one-time search.

Pass 2: Schema Mapping
Purpose:
Convert the semantic meaning into the fixed mission schema.

Example pass 2 output:
{
  "entity_category": "pubs",
  "location_text": "Arundel",
  "constraints": [
    {
      "type": "text_compare",
      "field": "name",
      "operator": "contains",
      "value": "swan",
      "hardness": "hard"
    }
  ],
  "mission_mode": "research_now"
}

Critical rules

1. The LLM must map user language into semantic meaning, not preserve raw phrasing.
2. For text compare requests like:
   - with swan in the name
   - have the word swan in the name
   - called swan
   - name includes swan
   the final comparison value must be:
   - swan
   not "swan in the name"
3. Regex must no longer be used to interpret user intent.
4. Regex may only be used for safe cleanup such as trimming whitespace or removing obvious trailing boilerplate after the LLM has already chosen the constraint type.
5. The final structured mission must validate against a deterministic schema.
6. If the LLM returns unsupported constraint types, fields, or operators, validation must fail safely.
7. Preserve observability:
   store or emit
   - raw user input
   - pass 1 semantic interpretation
   - pass 2 structured mission
8. Keep compatibility with existing planner inputs where possible, but prefer the new structured mission shape as the canonical output.

4. Deterministic schema validation

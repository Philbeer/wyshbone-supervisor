TITLE: Truthful verification UX (Option B) + fix bridgeRunToUI 400

We are not changing agent behaviour.
We are fixing UX truthfulness and a broken integration.

CONTEXT

Currently:

Enrichment completes

Results appear in the chat bubble

Long attribute verification (beer garden, live music) runs silently

Activity panel stays “Executing” for 1–2 minutes

bridgeRunToUI returns 400

This makes the system feel hung, even when it is correct.

FIX 1 — Truthful UX during verification (Option B)

Goal:
Users must understand that results are provisional while verification is running.

Required changes

Introduce a verification phase artefact

When enrichment completes and provisional results are ready:

Emit artefact: verification_pending

Payload:

leads_count

attributes_being_verified (eg ["beer_garden", "live_music"])

total_checks_expected (leads × attributes)

UI-visible status

As soon as verification_pending exists:

Chat bubble must show:

“Provisional results generated. Verifying beer garden and live music now…”

Results list is clearly marked Provisional

Completion semantics

Do NOT mark run complete until verification finishes

Final Delivery Summary must distinguish:

provisional matches

verified matches

unverifiable attributes

No silent work

While verification is running, the Activity panel is allowed to stay “Executing”

But the user must always see why

FIX 2 — Fix bridgeRunToUI 400 (correctness)

Goal:
bridgeRunToUI must succeed or fail loudly with diagnostics.

Required changes

Instrument bridgeRunToUI

Log:

runId

conversationId

request payload (sanitised)

response status + body

Prefix logs with [RUN_BRIDGE]

Fix the request contract

Ensure bridgeRunToUI sends the canonical runId

Ensure conversationId matches the UI session

Remove any stale or mismatched identifiers

Failure handling

If bridgeRunToUI fails:

Emit artefact: run_bridge_failed

Include status + error reason

Do NOT silently ignore

DEFINITION OF DONE

Users see “Verifying…” immediately after provisional results

No unexplained “hang” while Activity is executing

bridgeRunToUI no longer returns 400 in normal runs

If it fails, failure is visible via artefact and logs

No agent logic, Tower logic, or learning logic changed

IMPORTANT NON-GOALS

Do NOT parallelise verification yet

Do NOT change verification rules

Do NOT suppress verification

Do NOT add performance hacks

This is about truth, clarity, and correctness only.

You are adding QA truthfulness instrumentation to Wyshbone Supervisor.

IMPORTANT
Do NOT change agent behaviour.
Do NOT change planner decisions.
Do NOT change Tower verdict logic.

This task is observability only.

Problem

The QA dashboard needs truthful per-layer run status, but today too much meaning is inferred indirectly from the final Tower verdict.

We need Supervisor to emit a compact per-run execution summary artefact so the UI can display where the run actually succeeded or failed.

Goal

At the end of each run, emit a structured QA layer summary artefact.

What to implement

PART 1 — Add a new artefact type

Add a final diagnostic artefact for completed benchmark runs:

qa_layer_summary

If adding a new artefact type is risky, use an existing diagnostic artefact type but title it exactly:

QA Layer Summary

PART 2 — Artefact payload

The payload should include:

{
  query: string,
  benchmark_query: boolean,
  interpretation_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  planning_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  execution_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  discovery_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  delivery_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  verification_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  tower_status: "pass" | "fail" | "blocked" | "timeout" | "unknown",
  overall_outcome: "PASS" | "PARTIAL_SUCCESS" | "BLOCKED" | "TIMEOUT" | "FAIL",
  outcome_reason: string
}

PART 3 — Deterministic rules

Use simple deterministic rules.

Suggested rules:

interpretation_status
- pass if mission extraction / constraints extraction succeeded
- blocked if clarify gate blocked before execution
- fail if parsing/mission creation failed

planning_status
- pass if a mission plan was created
- fail if no valid plan exists

execution_status
- pass if at least one tool call or mission-driven execution step ran
- blocked if execution never started because of clarify/block
- timeout if run timed out

discovery_status
- pass if leads were found above zero
- fail if discovery ran and found none where results were expected

delivery_status
- pass if final delivery artefact exists with delivered leads > 0
- fail if execution completed but no delivery occurred

verification_status
- pass if any delivered leads carry verification evidence
- fail if delivery happened but zero leads carry verification evidence
- blocked if verification never became applicable because run blocked early

tower_status
- pass if Tower passed or accepted
- fail if Tower failed/stopped due to unmet requirements
- unknown if Tower never ran

overall_outcome
- PASS if intended test outcome achieved
- PARTIAL_SUCCESS if discovery and delivery passed but verification or Tower failed
- BLOCKED if clarify/block occurred before execution
- TIMEOUT if run timed out
- FAIL otherwise

PART 4 — Benchmark-only is acceptable

If safer, only emit this artefact for benchmark runs.

PART 5 — Logging

Also log the same summary in structured logs for easier debugging.

PART 6 — Deliverables

Provide:
1. files changed
2. where the artefact is emitted
3. the exact rules used
4. confirmation that this changed observability only, not behaviour

I have a follow-up fix needed after the sticky routing change. Two issues:
Issue 1: When chat-handler asks a SEARCH-SETUP question (e.g. "where would you like me to search?"), the sticky flag is being set as if it were a conversational question. The user's short answer ("sussex") then stays in chat-handler instead of kicking off the search with proper context.
Issue 2: When a search does eventually kick off from a conversational thread, it doesn't carry forward the topic context. E.g. user was chatting about "english sparkling wines" then asked to "find retailers in sussex" — the resulting search is for generic retailers, not wine retailers.
Fix 1 — don't set sticky flag for search-setup questions
In chat-handler.ts, after the LLM generates a response, detect whether the question is a search-setup question vs a conversational question. A search-setup question typically contains phrases like:

"where would you like me to search"
"what type of business"
"in which city"
"in what area"
"what sort of X are you looking for"

If the question looks like a search-setup question, DON'T set awaiting_chat_reply. The next user turn should go back through the router.
tsconst searchSetupPatterns = [
  /where.{0,30}(search|look|find)/i,
  /what (type|kind|sort) of/i,
  /which (city|town|area|region|location)/i,
  /in (what|which) (area|city|town|region)/i,
];
const isSearchSetupQuestion = searchSetupPatterns.some(re => re.test(response));

if (endsWithQuestion && !isSearchSetupQuestion) {
  // conversational follow-up — set sticky flag
  await setConversationFlag(conversationId, { awaiting_chat_reply: true, ... });
} else {
  await setConversationFlag(conversationId, { awaiting_chat_reply: false });
}
Fix 2 — pass chat context into search intent extraction
When the router decides to run a full search, the intent extractor needs the last 3-4 conversational turns so it can pick up qualifiers like "wine" from previous discussion.
Find the intent extractor call in supervisor.ts (or wherever the search path starts). Currently it probably only receives the current user message. Change it to pass:
tsconst recentContext = conversationHistory.slice(-4).map(m => `${m.role}: ${m.content}`).join('\n');
const extractorInput = {
  rawMessage,
  conversationContext: recentContext,
};
Then in the intent extractor's prompt, add an instruction like:
If CONVERSATIONAL CONTEXT is provided, use it to enrich the current search query with relevant qualifiers. 
Example: if recent context mentions "English sparkling wines" and the current message is "find retailers in Sussex", 
the extracted intent should be "wine retailers in Sussex" or "English sparkling wine retailers in Sussex", 
NOT just "retailers in Sussex".

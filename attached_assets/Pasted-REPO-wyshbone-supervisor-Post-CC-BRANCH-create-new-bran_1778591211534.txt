CONTEXT — READ THIS FIRST

The gpt4o_search executor returns leads stamped `verified=true` based purely on
GPT-4o's own confidence rating. Tower's per-constraint semantic judge is never
invoked. This is wrong: Tower must adjudicate per (lead × hard constraint), the
same way it does for the gp_cascade executor.

The gp_cascade reference pattern is in `server/supervisor/mission-executor.ts`
inside `processOneLead` (~line 1600–1800). For each lead, for each evidence
constraint, it calls `requestSemanticVerification(...)` from
`server/supervisor/tower-semantic-verify.ts`. That returns a TowerSemanticStatus
of 'verified' | 'weak_match' | 'no_evidence' | 'insufficient_evidence'. The
supervisor then stamps that onto a `constraint_led_evidence` artefact and an
in-memory EvidenceResult.

DO NOT emit artefacts with type 'attribute_evidence'. The supervisor does not
currently emit that type anywhere; emitting it would not connect to any live
path. Use `constraint_led_evidence` to mirror gp_cascade exactly.

DO NOT remove or weaken gpt4o_search's existing GPT-4o self-rating. It stays as
the belt-and-braces pre-filter. Tower's verdict overrides its claims.

DO NOT touch the planner metadata or which executor gets picked when.

DO NOT change the date bug in the GPT-4o search prompt (~line 187 of
gpt4o-search.ts: `Date.now() - 365 * 24 * 60 * 60 * 1000`). Park that for a
separate branch.
